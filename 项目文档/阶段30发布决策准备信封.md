- `release-approved`
- `releaseReady=true`
- `phase29ExitReady=true`

## 6. Release Owner 审阅条件

只有同时满足以下条件，信封才能从 `not-ready` 变为 `ready-for-human-release-review`：

1. live submission 已由维护者明确创建，并通过 `phase30:evidence-submission-gate` 格式校验。
2. 10 个 evidence slots 都有真实 reviewer、reviewedAt、evidenceRef、disposition、decisionReason 和 residualRisk。
3. `phase30:human-evidence-conflict-review` 没有 unresolved conflict。
4. `phase30:risk-disposition-followup-ledger` 对所有风险、延期和阻断项都有完整 followup 记录。
5. release owner 明确开始人工发布审阅。
6. 仍保持 `releaseReady=false`，直到后续独立 release approval review 明确批准。
7. 运行 `phase30:entry-readiness-redline`，确认 `gateCascadeAllowed=false`，并确认发布准备不会自动级联到 Phase 29 exit、Phase 30 entry、runtime go/no-go 或 third-party execution。
8. 使用 `phase30:release-approval-decision-packet` 准备 release owner 决策记录格式，但不得把准备包当作 release approval。

## 7. 禁止事项

- 不自动创建 `data/phase30-human-evidence-submission.json`。
- 不把 `ready-for-human-release-review` 当作 release approval。
- 不把信封完成当作 Phase 29 exit 或 Phase 30 entry。
- 不把信封完成当作 runtime go/no-go。
- 不把格式通过、冲突解决或风险跟踪完成自动转换为发布批准。
- 不设置 `releaseReady=true`。
- 不设置 `phase29ExitReady=true`。
- 不设置 `phase30EntryReady=true`。
- 不启用 `runtimeExecution`。
- 不启用 `thirdPartyExecution`。
- 不绕过 `阶段30入口准备红线.md` 中的独立 gate。
- 不把 `阶段30发布审批决策包.md` 当作 release owner 已经批准。

## 8. 完成判定

本信封完成时，只表示项目具备发布决策准备材料的汇总流程。它不代表 release owner 已经批准发布，不代表 live submission 已经创建，也不代表任何 evidence slot 已经通过人工审查。
