# 阶段 30 审计持久化演练

更新日期：2026-07-03

Baseline: `2.0.3 / phase30-audit-persistence-dry-run`

Phase 30 audit persistence dry-run: active

Audit persistence remains dry-run only.

## 1. 演练目标

`2.0.3 / phase30-audit-persistence-dry-run` 使用 `2.0.1 / phase30-runtime-sandbox-acceptance-harness` 和 `2.0.2 / phase30-secret-boundary-plan` 的 dry-run evidence 字段，定义审计持久化演练结构。

本演练只验证审计事件、关联 ID、导出摘要、复核占位和回滚线索的格式完整性，不写入真实运行时状态，不执行真实第三方插件，不开放 secret 或 SQLite 私人记忆读取。

本演练不改变当前工程状态：

- `PHASE=29`
- `APP_VERSION=1.9.48`
- `BUILD_LABEL=phase29-release-exit-final-archive-manifest-preview`
- `releaseReady=false`
- `phase29ExitReady=false`
- `phase30EntryReady=false`
- `runtimeExecution=false`
- `thirdPartyExecution=false`

No runtime state persistence in phase30-audit-persistence-dry-run.

## 2. 输入来源

审计 dry-run 只读取以下规划资产：

- `阶段30运行时沙箱验收框架.md`
- `阶段30密钥边界计划.md`
- `阶段30入口基线.md`
- Phase 29 runtime audit replay
- Phase 29 runtime evidence review
- Phase 29 release exit signoff evidence reconciliation preview

这些输入用于定义审计结构，不用于写入运行时状态。

## 3. 审计事件 Schema

每条 dry-run audit event 应包含：

- `auditEventId`
- `schemaVersion`
- `sourceBaseline`
- `fixtureId`
- `boundaryId`
- `decision`
- `observedResult`
- `expectedResult`
- `blockedReason`
- `quarantineRequired`
- `manualReviewRequired`
- `redactionApplied`
- `evidenceRef`
- `correlationId`
- `rollbackHint`
- `runtimeExecution=false`
- `thirdPartyExecution=false`
- `releaseReady=false`
- `phase30EntryReady=false`

审计事件不得包含 secret value、原始记忆正文、展品说明全文、附件原始内容、SQLite 数据行或签名私钥。

## 4. Dry-run Ledger

审计演练使用只读 ledger 设计：

| Ledger 字段 | 说明 | 当前策略 |
| --- | --- | --- |
| auditEventId | 单条审计事件 ID | required |
| correlationId | fixture、boundary、review 的关联 ID | required |
| sourceBaseline | `2.0.1` 或 `2.0.2` | required |
| decision | pass-dry-run、blocked、quarantined、manual-review-required | required |
| evidenceRef | 指向 dry-run evidence，不含敏感内容 | required |
| reviewerPlaceholder | 人工复核占位 | optional |
| rollbackHint | 未来回滚线索，不执行回滚 | required |
| persistedRuntimeMutation | 是否写入运行时状态 | always false |

Ledger 只能作为演练结构出现，不写入真实 runtime store。

## 5. 关联规则

Phase 30 audit dry-run 必须固定以下关联规则：

- 每个 fixture 至少有一个 `correlationId`。
- 每个 blocked 事件必须关联 `blockedReason`。
- 每个 quarantined 事件必须关联 `quarantineRequired=true`。
- 每个 manual-review-required 事件必须关联 reviewer placeholder。
- 每个 secret boundary 事件必须关联 `boundaryId`。
- 每个 pass-dry-run 事件必须保留 `runtimeExecution=false` 和 `thirdPartyExecution=false`。
- 每个 audit export summary 都必须保留 redaction summary。

## 6. 导出摘要

dry-run audit export summary 应包含：

- `schemaVersion`
- `baseline`
- `eventCount`
- `blockedCount`
- `quarantinedCount`
- `passDryRunCount`
- `manualReviewRequiredCount`
- `redactionSummary`
- `runtimeMutationCount=0`
- `secretValueCount=0`
- `privateMemoryPayloadCount=0`
- `releaseReady=false`
- `phase30EntryReady=false`

导出摘要可以用于后续 `2.0.4 / phase30-runtime-go-no-go-board`，但不能作为 release approval 或 runtime enablement 的自动依据。

## 7. 禁止事项

审计 dry-run 明确禁止：

- persist-runtime-state
- persist-release-state
- persist-phase30-entry-state
- persist-secret-value
- persist-private-memory-payload
- execute-third-party-plugin
- mark-releaseReady-true
- mark-phase30EntryReady-true
- convert-blocked-to-pass-without-review
- use-audit-export-as-release-approval
