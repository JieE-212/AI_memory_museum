## 1. 项目目标

AI 记忆博物馆是一个娱乐型 Agent 项目。用户可以输入日记、聊天片段、照片描述、旅行片段、梦境或人生片段，系统会把它整理成一件“私人博物馆展品”。

项目当前已经实现：

- 用户添加记忆
- AI 自动生成展品标题
- AI 自动生成展品说明
- AI 判断所属展厅
- AI 提取人物、地点、时间、情绪和标签
- 用户可以按展厅参观记忆
- SQLite 持久化保存展品
- AI 讲解员可以根据用户的记忆回答问题
- 简单 RAG 检索候选展品
- hybrid RAG 检索候选展品，并展示命中字段、语义线索和引用依据
- semantic/keyword/hybrid 三种检索模式已有 smoke 覆盖
- 检索引用会展示强证据、可参考、弱关联、弱证据等可信度层级
- 讲解员在未找到强匹配展品时会说明证据边界
- 讲解员会根据召回结果给出可继续追问方向
- 添加记忆区域右侧已有轻量展品草稿预览，减少桌面端空白
- 第九阶段第一版支持封面图线索、图片/OCR/语音说明和附件清单元数据
- 多 Agent 状态机展示整理过程
- 人工确认 Agent 步骤和复核项
- Agent 运行摘要和最近动作预览
- Agent run、step、event 的第七阶段持久化结构

项目后续继续增强：

- 更强的语义检索、混合 RAG 召回和讲解依据展示
- 更完整的 Agent 审计日志查询、清理策略和跨展品整理历史回放
- 多模态记忆展品、照片封面、OCR 和附件线索
- 时间线、主题展、年度回忆报告和跨展品洞察
- 账号、多端同步、隐私保护、导出删除和部署能力
- 更完整的自动化测试和可维护工程结构
- 必要时评估 LangGraph.js 管理多 Agent 编排

## 2. 项目核心体验

用户不是在使用普通聊天机器人，而是在建造一座关于自己的虚拟博物馆。

一段普通记忆会被整理成：

- 展品标题
- 原始记忆
- 展品说明
- 所属展厅
- 情绪标签
- 关键词标签
- 相关人物
- 地点和时间
- 记忆来源
- 珍藏级别
- 是否重点展品

第六阶段新增的核心体验是：用户不只可以看到 Agent 工作流如何一步步整理记忆，还可以确认某个 Agent 步骤、补全复核项、确认复核完成，必要时触发重试整理。

## 3. 阶段规划

### 阶段 1：纯前端 Mock 版

目标：先做出可见、可玩的产品原型。

功能：

- 添加文字记忆
- 展示记忆卡片
- 手动或 Mock AI 生成展品信息
- 按展厅筛选
- 按情绪和重点展品筛选
- 搜索展品
- 查看详情
- 编辑展品
- 展品排序
- JSON 导入导出备份
- 结构化 JSON 样例预览
- 数据规范化和基础安全渲染
- 数据保存到浏览器 localStorage

技术：

- HTML
- CSS
- JavaScript
- localStorage

当前状态：已完成。第一阶段已经补充安全渲染、编辑展品、排序、带 schema 的 JSON 导入导出、结构化 JSON 样例预览、更丰富的情绪体系、情绪强度、记忆来源、珍藏级别、重点展品、随机示例、清空筛选、输入长度控制、导入数据规范化和完整的空状态/响应式细节。

### 阶段 2：加入真实 AI 生成

目标：把 Mock AI 替换为真实 AI API。

功能：

- 用户输入原始记忆
- 后端调用 AI
- AI 返回 JSON 格式的结构化展品信息
- 前端展示并允许用户修改
- 未配置 API Key 或调用失败时自动回退 Mock
- 提供后端健康检查接口
- 提供 schema/options 查询接口
- AI 请求超时控制和错误码规范化
- 前端 AI 请求状态反馈
- 原始记忆字数反馈
- 通过 `.env` 配置 OpenAI-compatible API

技术：

- Node.js
- Node.js HTTP 服务
- AI API
- Prompt 设计

当前状态：已完成。项目通过 `npm.cmd start` 启动 Node 后端，后端负责托管静态页面并提供 `/api/health`、`/api/options` 和 `/api/analyze`。`/api/analyze` 会调用 OpenAI-compatible Chat Completions API，要求 AI 返回结构化 JSON；没有配置 `AI_API_KEY`、调用超时或调用失败时，会自动使用 Mock 分析回退，保证项目可运行。

### 阶段 3：加入数据库

目标：让记忆不只保存在浏览器，而是保存在数据库中。

数据库：SQLite。

主要数据表：

- memories：记忆展品表
- halls：展厅表
- memory_people：展品人物关联表
- memory_tags：展品标签关联表
- memory_emotions：展品情绪关联表

详细设计见 `项目文档/第三阶段数据库设计.md`。

当前状态：已完成。后端使用 Node 内置 `node:sqlite` 自动创建 `data/memory-museum.sqlite`，并提供 `/api/memories` CRUD、`/api/memories/import`、`/api/memories/export` 和 `/api/search`。前端已改为数据库优先保存，后端不可用时保留 localStorage 本地备份；页面提供“同步数据库”和“迁移本地”入口，用于把旧浏览器数据写入 SQLite。导入接口已支持整批事务和 ID 冲突保护，数据库/本地备份不一致时会合并显示并提示迁移。

### 阶段 4：加入简单 RAG

目标：AI 讲解员可以基于用户保存的记忆回答问题。

功能：

- `GET /api/search?query=关键词` 从数据库中检索候选展品
- `POST /api/guide` 抽取问题关键词、检索候选展品并生成讲解员回答
- 按标题、正文、标签、情绪、人物、来源等字段加权检索
- 未配置 AI Key 时使用本地 RAG 回退回答
- 前端展示候选依据、检索词、检索模式和 fallback 提示

当前状态：已完成。第四阶段已经完成收尾优化，并成为第五阶段讲解员 Agent 的基础能力。

### 阶段 5：加入 Agent 工作流

目标：让系统从“AI 生成器”升级为“Agent 系统”。

当前角色：

- 档案员 Agent：提取结构化信息
- 策展人 Agent：判断展厅分类、情绪和珍藏级别
- 编辑 Agent：生成展品标题、标签和说明
- 讲解员 Agent：整理导览提示，并在保存后参与 RAG 问答

当前已实现：

- `POST /api/analyze` 返回结构化展品草稿、`workflow.summary` 和 `workflow.steps`
- 工作流拆成档案员 Agent、策展人 Agent、编辑 Agent、讲解员 Agent 四步
- summary 包含总步数、已完成数、就绪数、运行状态、进度、下一步建议和复核项
- 前端展示 Agent 工作流轨迹、进度条、状态标签和复核区
- 复核项会标明对应字段，并可点击定位到人物、地点、时间、标签或展品说明
- 用户补完复核字段后，复核项会自动隐藏，并显示“复核完成，可以保存”
- 工作流仍在运行且暂无复核项时，会显示“等待 Agent 输出”，不会误显示“无需复核”
- 用户可以重新应用 Agent 建议，也可以收起工作流
- 未配置 `AI_API_KEY` 或调用失败时，后端和前端都会保留 Mock Agent 工作流回退
- 页面已围绕“记忆博物馆”做多轮温馨化优化，包括记忆图案、展品卡片、馆藏编号、空状态和响应式细节

当前状态：第五阶段已经完成多轮修复、优化和迭代，并已作为第六阶段的稳定基础。显式工作流已经在第六阶段升级为带状态机元数据、人工确认和重试入口的可控流程。

### 阶段 6：Agent 状态机和人工确认

目标：让 Agent 工作流从“展示轨迹”升级为“可控流程”。

第一版已实现：

- 为每个 Agent 步骤增加明确状态：queued、running、needs_review、approved、failed、done
- `/api/analyze` 返回 `workflow.stateMachine`、`workflow.summary.confirmationItems` 和更明确的 step actions
- 支持用户确认 Agent 步骤
- 支持待确认步骤触发重试整理入口
- 支持在保存前显示并确认复核清单
- 复核项补完后需要点击“确认复核完成”，确认后才进入可保存状态
- 保存前会校验复核确认状态，避免绕过人工确认直接保存
- Agent 步骤会展示依据字段，方便观察每一步参考了哪些结构化线索
- Agent 工作流会展示 run id、阶段、事件数量和最近动作，为第七阶段审计日志做准备
- 页面会展示 Agent 状态机轨道，让 queued、running、needs_review、approved、failed、done、ready 的当前位置更清楚
- 第六阶段 readiness 检查曾覆盖状态机、人工确认、保存闸门和 `/api/analyze` 第六阶段接口结构
- 当前项目已升级为第十阶段 readiness 检查，继续保留这些状态机回归点

后续增强：

- 支持用户驳回 Agent 给出的关键字段
- 支持只重试某一个 Agent 步骤，并保留其他步骤结果
- 支持记录 Agent 生成过程中的依据、提示和修改历史
- 支持更细的错误提示，例如 AI 调用失败、JSON 解析失败、字段缺失、数据库保存失败
- 优化讲解员对已保存展品的引用方式，让回答更可追溯

技术方向：

- 先用当前原生 Node.js + JavaScript 扩展状态机
- 如果流程复杂度上升，再评估 LangGraph.js
- 补充核心接口和状态转换的测试用例

第五阶段进入第六阶段的标准：

- 第五阶段工作流稳定可用
- 复核项和保存流程没有明显状态误导
- 前端布局在桌面和移动端都可用
- `/api/analyze`、`/api/guide`、数据库 CRUD 和导入导出接口保持可回归

阶段 6 收尾优化：

- `/api/analyze` 已返回轻量 `workflow.run` 元数据
- 前端已展示 Agent 运行摘要和最近动作
- 确认本步、重试整理、确认复核、重新应用建议都会形成前端事件预览
- readiness 检查已覆盖 run 元数据、审计预览 UI 和 API smoke 结构

当前结论：第六阶段状态机、保存闸门、API smoke test 和数据库迁移策略已经稳定，且已经补齐第七阶段所需的 run/step/event 概念基础；项目已经进入阶段 7。

### 阶段 7：Agent 运行历史和审计日志

目标：把第六阶段的“前端可见状态机”升级为“后端可持久化、可查询、可关联展品、可审计的 Agent 运行记录”。

第一版计划功能：

- 为每次 `/api/analyze` 创建唯一 `runId`
- 持久化 Agent run、step、状态变更、重试次数、确认人机操作和失败原因
- 展品保存时关联最近一次 Agent run
- 展品保存时会同步当前工作流快照，保留用户确认、重试和复核事件
- 用户可以驳回单个 Agent 步骤，`step_rejected` 会进入整理历史
- 展品保存动作会写入 `memory_saved` 审计事件，并区分创建和更新
- 展品详情页展示“整理来源”，可查看运行进度、原始线索、事件数量、依据字段和四个 Agent 步骤摘要
- 首屏统计增加整理历史数量，优先使用后端 health stats 中的真实 Agent run 计数
- 页面布局继续向温馨私人博物馆优化，包括更柔和的首屏文案、保存历史提示和展品卡片历史标识
- 提供 `/api/agent-runs/:id` 查询运行详情
- 提供 `/api/memories/:id/agent-run` 查询某件展品的整理历史
- API smoke test 覆盖 run 创建、step 保存、event 保存、确认/驳回事件、创建/更新保存事件审计、无效 run 清理和展品关联

后续增强：

- 后续可继续细化驳回理由输入和驳回后的局部再生成
- 支持只重试单个步骤，保留其他步骤结果和人工修改
- 增加 Agent run 清理策略，避免本地数据库无限增长

技术方向：

- 新增 `agent_runs`、`agent_steps`、`agent_events` 数据表
- `/api/analyze` 返回并保存 run 级别元数据
- `/api/agent-runs/:id` 查询运行详情
- 状态转换规则先用原生 JavaScript 明确实现
- readiness 检查升级为第七阶段检查脚本

已满足的进入标准：

- 第六阶段状态机、保存闸门和 API smoke test 稳定通过
- 前端 Agent 面板没有明显状态误导
- 数据库迁移策略清晰，不破坏已有展品数据

完成标准：

- 重新整理、确认、保存、驳回和重试都有可追溯记录
- 展品详情能展示本展品的 Agent 整理历史
- 临时后端 smoke test 能验证 run 创建、step 状态和事件记录

当前结论：以上完成标准已经满足。第七阶段可以作为阶段 8“语义检索和混合 RAG”的稳定基础。

### 阶段 8：语义检索和混合 RAG

目标：让 AI 讲解员从关键词召回升级为语义理解，能回答更模糊、更情绪化、更跨展品的问题。

第一版已实现：

- `/api/search` 支持 `mode=keyword|semantic|hybrid`
- hybrid 检索会结合关键词和本地语义线索扩展；semantic 模式会保留原始查询锚点，避免只看扩展词导致漏召回
- 检索结果返回 score、matchedTerms、semanticTerms、matchedFields、reason 和 confidence
- `/api/guide` 默认使用 hybrid 检索，并返回 citations 和 followUps
- 前端讲解员展示候选展品、引用依据、命中字段、语义线索、匹配原因、可信度、继续追问建议和检索模式
- 添加记忆区域右侧展示实时草稿预览和入馆线索，为阶段 9 的封面图、附件和 OCR 线索预留位置
- mock 讲解员回答会展示前两条引用依据和可信度；未找到强匹配时会说明“证据边界”
- 未配置 embedding 服务时继续保留本地检索和关键词回退

后续计划功能：

- 为展品标题、说明、原文、标签、人物和情绪生成检索文本
- 支持 embedding 向量写入本地存储或轻量向量索引
- `/api/guide` 继续增强引用展品、命中字段、匹配理由和可信度
- 支持“只在我的记忆中回答”的回答边界
- 未配置 embedding 服务时保留关键词检索回退

技术方向：

- 优先保持 SQLite 主存储，可评估 sqlite-vss、向量扩展或本地 JSON 向量索引
- 增加 embedding 生成队列，避免保存展品时阻塞页面
- RAG prompt 增加引用格式和“不知道就说明没有找到”的约束
- 增加搜索质量样例测试

进入标准：

- 阶段 7 已经能追踪讲解员召回依据
- 展品保存和导入导出流程稳定
- 本地回退检索仍能独立工作

完成标准：

- 用户用模糊问题也能召回相关展品
- 讲解员回答明确展示引用依据
- 无 AI Key 或 embedding 不可用时仍可使用关键词 RAG

当前结论：以上完成标准已经满足。第八阶段可以作为阶段 9“多模态记忆和附件展品”的稳定基础。

### 阶段 9：多模态记忆和附件展品

目标：让记忆博物馆从文字展品扩展为包含照片、截图、语音转写和附件线索的私人档案空间。

计划功能：

- 支持为展品添加封面图线索和附件清单
- 支持图片/OCR/语音线索手动录入，并进入保存、导入导出和检索
- 支持图片预览、替换、删除和导出
- 支持 OCR 提取截图或照片中的文字线索
- 支持图片说明生成，作为 Agent 整理输入的一部分
- 支持语音转文字后进入展品整理流程
- 展品墙支持更温馨的图片卡片布局和无图占位图案

第一版已实现：

- `memories` 表新增 `cover_image`、`media_note` 和 `attachments_json`
- 前端添加封面图线索、图片/OCR/语音线索、附件清单输入
- 展品卡片展示封面线索和附件数量提示
- 展品详情展示多模态线索和附件清单
- 搜索会命中封面、媒体说明和附件字段
- 导入导出会保留多模态元数据
- 页面明确提示当前保存的是附件清单和线索，原文件上传能力后续接入
- 首屏会统计带多模态线索的展品数量
- 展品墙支持直接筛选多模态线索展品
- 导出文件包含 `mediaSummary`，用于检查封面、媒体说明和附件清单覆盖情况

## 5. 技术路线

第一阶段：

- 静态 HTML 页面
- CSS 响应式布局
- JavaScript 管理数据
- localStorage 本地保存

第二阶段：

- Node.js 后端托管静态页面
- 后端 AI 分析 API
- OpenAI-compatible Chat Completions
- `.env` 管理 AI 配置
- 前端请求后端生成结构化展品草稿

第三阶段到第五阶段：

- 数据库已使用 SQLite
- 后端已提供 `/api/memories` CRUD、导入导出和搜索接口
- RAG 已通过 `/api/search` 和 `/api/guide` 接入
- Agent 工作流已通过 `/api/analyze` 返回 `workflow.summary` 和 `workflow.steps`
- Agent 状态机已通过 `/api/analyze` 返回 `workflow.stateMachine`、`needs_review` 和 `approved` 等状态
- AI 接口使用 OpenAI API 或兼容 OpenAI 格式的模型服务
- `npm.cmd run check` 已包含语法检查、第十五阶段 readiness 检查和 API smoke test
- `npm.cmd run smoke` 可单独执行接口 smoke test

第八阶段检查基线：

- `scripts/phase15-readiness.js`：静态检查第十五阶段状态、专题资产表、报告草稿表、资产 API、导出字段、前端阶段期望、API smoke 覆盖、CSS 变量、README、白皮书和项目规划状态
- `scripts/api-smoke.js`：临时启动后端和临时 SQLite，验证 `/api/health`、`/api/version`、`/api/operations`、`/api/operations/export`、`X-Request-Id`、`/api/privacy`、`/api/workflows`、`phase12Readiness`、`/api/analyze`、`workflow.stateMachine`、`workflow.run`、`/api/memories/purge`、`/api/search?mode=keyword|semantic|hybrid`、`/api/guide`、`/api/insights`、`/api/agent-runs/:id`、`/api/memories/:id/agent-run`、`confirmationItems` 和 step evidence

阶段 7+ 技术路线：

- Agent run 先以 SQLite 表持久化，保留本地优先和易备份特性
- 前端保持原生 JavaScript 技术栈，同时把后端 health stats 纳入页面统计，减少重复估算
- RAG 从关键词检索逐步升级为语义检索和混合检索，但必须保留无 AI Key 回退
- 多模态能力先做成可选增强，图片、OCR、语音转写都要允许手动替代
- 时间线、主题展和报告优先复用已有展品字段，再补充必要的统计 API
- 工作流复杂到需要暂停、恢复和分支时，再评估 LangGraph.js 或自建轻量编排层
- 前端是否升级 React + Vite，应以页面复杂度和维护收益为依据
- 后端是否升级 Express，应以路由复杂度、测试便利性和中间件需求为依据
- 阶段 13 前持续补充自动化测试、数据迁移检查和文档化启动流程
