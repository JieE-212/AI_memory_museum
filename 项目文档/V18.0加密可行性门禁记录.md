# V18.0 加密可行性门禁记录

> 记录日期：2026-07-21
>
> 当前结论：**NO-GO——不得进入 V18.1，也不得把现有项目称为“真正的加密锁馆”**
>
> 范围说明：本文只记录本地可行性验证；不代表代码已提交、推送、部署或发布。

## 先说人话：为什么 Windows 测通了，仍然不能继续

本次实验已经证明：在当前 Windows x64 开发机上，一个固定版本的原生 SQLite3MC 驱动能够创建无法被普通 SQLite 直接读取的加密数据库，并能继续使用时屿依赖的 WAL、全文检索、JSON、外键和事务能力。

但时屿的实际部署目标还包括基于 Linux `node:24-bookworm-slim` 的容器和 CloudBase。当前机器没有可用的 Docker 或 WSL 环境，因此没有完成相同驱动在 Linux / CloudBase 等效环境中的安装、加载和完整 canary。加密属于“一个目标环境没证明，就不能默认安全”的能力，所以总体门禁保持 NO-GO。

这不是功能失败，而是主动停在证据边界：宁可明确说“尚未证明”，也不把一次 Windows 成功包装成跨平台完成。

## 固定实验环境

| 项目 | 已验证值 |
| --- | --- |
| 操作系统与架构 | Windows x64（`win32-x64`） |
| Node.js | `v24.15.0` |
| Node ABI | `137` |
| 驱动包 | `better-sqlite3-multiple-ciphers@12.11.1` |
| 原生模块 | `better_sqlite3.node`，2,280,960 bytes |
| 原生模块 SHA-256 | `8E0CC725ADE3D9BB0E02D10DD082BAAC1107014A1BEA8E0AF2106C48828847A8` |
| SQLite | `3.53.2` |
| SQLite3 Multiple Ciphers | `2.3.5` |

版本与哈希被固定，是为了让面试官或后续维护者知道“验证的究竟是哪一个二进制”，而不是只看到一个随时可能变化的包名。该驱动未作为 V18 生产依赖接入项目；canary 在隔离临时目录中运行。

## Windows canary 已经证明的内容

以下是已实际执行并通过的场景，不用虚构的断言数量包装结果：

| 验证项 | 结果 | 它说明什么 |
| --- | --- | --- |
| 加密后文件头检查 | 通过 | 数据库文件不再以明文 SQLite 的 `SQLite format 3` 文件头出现。 |
| 明文 canary 扫描 | 通过 | 写入数据库的特征明文没有在加密数据库字节中直接检出。 |
| 错误口令打开 | 通过 | 使用错误密钥不能正常读取受保护数据库。 |
| 正确口令重开 | 通过 | 关闭数据库后，用正确密钥可以重新打开并读回预期数据。 |
| WAL 工作模式 | 通过 | 加密连接可以使用项目依赖的 WAL 工作流。 |
| FTS5 | 通过 | 加密连接仍可创建并查询全文检索能力。 |
| JSON | 通过 | 现有数据层依赖的 JSON 查询能力可用。 |
| 外键 | 通过 | 外键约束可启用并按预期拒绝无效关系。 |
| 事务 | 通过 | 提交与回滚行为可用，未因加密驱动破坏原有事务语义。 |

这些结果证明的是“Windows 上这一个固定原生二进制的最低兼容性”，不是完整 V18 设计已经实现。媒体 AES-256-GCM、密钥派生与包装、vault worker、迁移回滚、加密备份以及小白向导仍未进入实现阶段。

## 工程卫生与临时数据清理

canary 使用的一次性目录名为 `time-isle-v18-native-canary-019f5ff7`。实验结束后已再次检查，该目录不存在；实验数据库、测试密钥、临时安装和特征明文没有保留在项目目录中。

本记录保留可复核的环境、哈希、场景和结论，但不保留以后不会使用的临时数据库、安装目录或一次性探针代码。项目依赖也没有因为实验被静默替换。

## 当前阻断项

要把门禁改为 GO，至少还缺下面这组同等证据：

1. 在项目 Dockerfile 对应的 Linux `node:24-bookworm-slim` 环境中，固定同一 npm 包版本，记录实际加载的 Linux 原生二进制 SHA-256、Node ABI、SQLite 和 SQLite3MC 版本。
2. 在该 Linux 环境重新执行明文文件头、明文 canary、错钥拒绝、正确钥重开、WAL、FTS5、JSON、外键和事务验证。
3. 证明驱动能够在 CloudBase 实际构建与运行约束下安装、启动和重开，而不只是“理论上支持 Linux”。
4. 将跨平台 canary 变成可长期运行的永久门禁，并确认失败时不会回退到明文数据库。

当前环境没有 Docker，也没有 WSL，因此以上证据尚未取得。仅凭 Windows 结果继续改造正式数据层，会把“构建失败”和“错误加密实现”推迟到部署或恢复阶段，风险不可接受。

## 名称与对外表述红线

本次候选驱动使用的是 **SQLite3 Multiple Ciphers（SQLite3MC）**。它提供与 SQLCipher 风格相近的加密能力，但不是 Zetetic 官方 SQLCipher；项目文档、界面、简历和面试介绍都不能把它写成“官方 SQLCipher”。

在门禁转为 GO 且 V18.1–V18.4 真正完成前，只能准确地说：

> V18 完成了 Windows 原生加密驱动的可行性 canary，但由于缺少 Linux / CloudBase 同等执行证据，项目主动停在 V18.0 NO-GO；当前发布版的锁馆仍是应用层写门禁，不是静态数据加密。

以下说法均不成立：

- “V18 已完成”或“真正的加密锁馆已上线”；
- “数据库已经使用官方 SQLCipher”；
- “Windows 测过，所以 CloudBase 一定可用”；
- “现有 V14 写门禁等于磁盘加密”；
- 用前端 AES、字段级零散加密或未验证驱动绕过本门禁。

## 门禁决定

**V18.0 总体结论：NO-GO。**

- 保留 Windows canary 作为正向可行性证据；
- 不把实验驱动接入正式数据层；
- 不进入 V18.1 保险库运行时；
- 不提交、不推送、不发布任何“V18 已完成”的版本；
- 等取得 Linux `node:24-bookworm-slim` 与 CloudBase 同等执行证明后，再重新召开 V18.0 门禁。
