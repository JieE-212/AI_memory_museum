# 阶段 30 收口审查包

更新日期：2026-07-05

Baseline: `2.0.7 / phase30-closure-review-package`

Phase 30 closure review package: active

本收口审查包用于把当前项目从“继续堆叠规划预览”切换到“可交付审查”。它不批准 release、不退出 Phase 29、不进入 Phase 30 implementation、不启用 runtime execution，也不允许第三方插件执行。

## 1. 审查目标

当前项目已经具备 Phase 29 冻结、Phase 30 规划基线、沙箱验收框架、密钥边界计划、审计持久化演练、Go/No-Go 看板、第一轮规划收口、人工复核证据包和人工复核执行台账。

下一步不再新增 Phase 29 preview artifact，也不继续创建新的运行时能力模型。优先完成：

- git 目录扁平化形态确认与索引迁移授权；
- release candidate 边界说明；
- 人工复核证据执行清单；
- 工程卫生和编码清理验证；
- 保持所有 runtime 与 release guardrail 关闭。

## 3. Git 形态审查

当前 `项目工程` 是实际 git 仓库根目录。人工已经确认接受当前扁平化目录形态，决策记录见 `仓库扁平化确认记录.md`。

索引仍保留旧的一层目录路径，表现为旧的 `项目工程/...` 路径被标记删除，而当前根目录文件被标记为未跟踪。路径迁移提交仍需后续人工授权。

在迁移提交授权前：

- 不执行 `git reset`；
- 不执行 `git checkout --`；
- 不执行 `git clean`；
- 不执行 `git add -A`；
- 不制作正式 release commit。

当前决策：

- `gitPathShape=accepted-flattened-project-root`
- `gitIndexMigration=committed-flattened-project-root`
- `releaseCommit=blocked-until-human-release-approval`

## 4. Release Candidate 边界

当前可以作为 release candidate 审查输入，但不能作为正式 release：

- `npm.cmd run check` 必须通过；
- `/api/version`、`/api/operations`、`/api/operations/export` 必须继续暴露 Phase 30 human review execution ledger；
- 10 个人工证据槽位必须继续是 `pending` 和 `evidenceRef=missing`；
- runtime 和 third-party execution 必须继续关闭；
- 工程卫生记录必须说明 git 目录扁平化已被接受，但索引迁移尚未完成。

RC 审查说明必须明确：

- 当前是 Phase 29 frozen review state；
- Phase 30 entry 尚未激活；
- 任何 check pass 都不能转换为人工批准；
- 任何 dry-run pass 都不能转换为运行时启用。

- release、Phase 29 exit、Phase 30 entry、runtime execution 和 third-party execution 仍保持阻断。

Phase 30 closure review package is not release approval.

## 8. 仓库索引迁移预检

本轮补充 `仓库索引迁移预检清单.md` 和 `git:flattening-preflight`，用于在真正迁移 git index 前做只读检查。

当前结论保持不变：

- `gitPathShape=accepted-flattened-project-root`
- `gitIndexMigration=committed-flattened-project-root`
- `indexMutation=committed-path-migration`

预检脚本只允许读取 git status，并确认旧索引路径删除、当前根目录未跟踪文件、忽略运行产物三类现象仍存在。预检通过不等于迁移授权，也不等于 release approval、Phase 29 exit、Phase 30 entry 或 runtime enablement。

## 9. 索引迁移执行方案边界

本轮补充 `仓库索引迁移执行方案.md` 和 `git:flattening-migration-plan`，用于把后续迁移拆成可审查对象，而不是直接执行。

新增关键边界：

- `docsOwnershipDecision=docs-in-repo`
- `indexMutation=committed-path-migration`
- `项目文档=outside-current-git-root`

结论：工程代码路径可以在后续授权后进入路径扁平化迁移准备，但旧索引中的 `项目文档/...` 需要先决定文档继续留在仓库外，还是迁回当前 git root 内。该决策完成前，不应制作路径迁移 commit。

## 10. 文档归属决策

本轮接受 `docs-in-repo`，并新增 `文档归属决策记录.md` 与 `docs:ownership-check`。

当前边界：

- `docsOwnershipDecision=docs-in-repo`
- `outsideMirrorRetained=true`
- `gitIndexMigration=committed-flattened-project-root`
- `indexMutation=committed-path-migration`

仓库内 `项目文档/` 作为后续版本控制位置；仓库外 `../项目文档` 暂时保留为镜像来源。本决策不等于 git index 迁移授权，也不等于 release approval、Phase 29 exit、Phase 30 entry 或 runtime enablement。

## 11. Git index 路径迁移提交检查

本轮新增 `git:flattening-index-staged-check`。该检查用于确认旧的 `项目工程/...` 索引路径已从 index 中移除，当前根目录文件与 `项目文档/` 已完成路径迁移提交。

当前边界：

- `gitIndexMigration=committed-flattened-project-root`
- `indexMutation=committed-path-migration`
- `commitCreated=true`
- `releaseCommit=blocked-until-human-release-approval`

这仍不代表 release approval、Phase 29 exit、Phase 30 entry、runtime execution 或 third-party execution。
