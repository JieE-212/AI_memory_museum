# 技术设计

## 架构概览

```text
Browser
  ├─ public/index.html
  ├─ public/styles.css
  └─ public/assets/app.js
            │ fetch JSON
            ▼
server.js / api/index.js
  ├─ 记忆 CRUD
  ├─ AI 整理与 Mock fallback
  ├─ 混合检索、讲解与回顾
  ├─ 导入导出与隐私边界
  └─ Demo 安全限制
            │
            ▼
database.js → SQLite
```

本地运行由 `server.js` 启动 Node HTTP Server；Vercel 环境由 `api/index.js` 导出同一个 request handler，避免维护两套路由。

## 数据模型

主要表：

- `memories`：原文、展品说明、展厅、日期、地点、重要度和媒体备注。
- `memory_people`、`memory_tags`、`memory_emotions`：多值字段，使用外键关联展品。
- `agent_runs`：一次整理的模式、状态、摘要和关联展品。
- `agent_steps`：档案、策展、编辑等步骤的输出与证据。
- `agent_events`：工作流开始、结构化、等待确认等事件。

升级已有 SQLite 时，旧版专题展和报告草稿表可能仍留在文件中，但 v2 不再读取或写入，也不会在新数据库中创建。为避免擅自销毁历史数据，升级过程不主动删除这些旧表。

## 记忆整理

`POST /api/analyze` 的处理顺序：

1. 限制原始文本长度并校验输入。
2. 有 `AI_API_KEY` 时请求 OpenAI-compatible Chat Completions。
3. 验证模型 JSON，并把字段归一化到允许的展厅、范围和长度。
4. 未配置 Key 或请求失败时，使用本地关键词规则生成草稿。
5. 创建三步 Agent workflow，并先保存 run。
6. 前端允许用户修改；确认后 `POST /api/memories` 保存展品并关联 run。

模型失败不会阻断核心流程，也不会把错误响应直接写入馆藏。

## 检索与讲解

SQLite 层读取馆藏后，对下列字段进行加权匹配：

- 标题、展品说明、原始记忆。
- 人物、地点、标签、情绪和来源。
- 少量中文语义词组扩展。

搜索结果返回分数、命中字段、命中词和置信提示。讲解员只接收排名靠前的展品摘要，并把相同展品作为 citations 返回前端。

这是一种适合小型个人馆藏的可解释检索方案，不把它包装成生产级向量 RAG。

## 数据安全

- `.env`、`data/`、日志和调试导出均被 Git 忽略。
- 完整导出包含原文；脱敏导出会隐藏原文、人物、地点、附件和媒体备注。
- 清空本地数据库需要正文 `DELETE` 确认。
- Demo 模式会阻止所有 `DELETE` 请求和清空接口。
- 响应包含 CSP、`nosniff`、禁止 iframe 和权限限制等基础安全头。

## Demo 模式

启用 `INTERVIEW_DEMO=true` 后：

- 数据库位置切换到系统临时目录，或使用显式 `DB_PATH`。
- 冷启动时注入四件示例展品和一条 Agent run。
- 删除与清空返回 403。
- 未配置真实 Key 时使用 Mock fallback。

Vercel `/tmp` 是临时且可能多实例的，因此 Demo 只用于功能体验，不作为真实用户数据存储。

## 测试策略

`scripts/frontend-check.js` 检查：

- 四个主视图和核心操作是否存在。
- 页面是否重新出现 Phase/Reviewer/插件治理术语。
- 是否存在内联脚本、渐变或失控的核心文件规模。
- 前端和服务端是否仍覆盖核心 API。

`scripts/api-smoke.js` 使用临时 SQLite 启动真实服务，覆盖：

- 静态资源与安全头。
- Mock 整理、Agent run 保存与展品关联。
- 展品 CRUD、检索、讲解、回顾、导入导出。
- 清空确认。
- Demo 示例注入、临时存储和删除保护。

## 扩展方向

只有在核心路径稳定且出现真实需求后，再考虑：

- 用户认证与托管数据库。
- 向量检索或 embedding 缓存。
- 图片和语音的受控上传。
- 可选择的端到端加密同步。

这些能力不应重新进入默认首页，也不应以阶段编号替代用户价值。
