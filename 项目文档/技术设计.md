# 技术设计：V14.0 共忆、设备内镜片与锁馆恢复验真（已发布）

## 设计目标

V14 延续“不增加顶层导航、本地优先、渐进披露”的产品边界，把 V4–V10 的影像、策展、回访、来源校准与受限 Agent，以及后续共忆见证、设备内镜片和写入保护，收束为可回看、可迁移且尊重用户明确判断的一套记忆系统。技术目标有二十七个：

1. 图片从不可信输入开始，经过格式验证、隐私派生、内容寻址、关联和回收的完整生命周期。
2. EXIF、相似检索、OCR 和图片区域都使用可追踪状态，候选与用户确认事实严格分离。
3. 主题展览、今日回访、实体线索与声音文字稿都保留来源、确认和待复核状态，不把推断冒充事实。
4. 时间胶囊只允许来自已发布且引用有效的主题展览；未到期读取只返回外壳，服务端不查询 payload。
5. 离线分享先经过公开外壳、密文内容、最终收据三层核对；章节、展品、引用、确认文字稿与安全展示图默认全部不选，投影结构物理排除未选原文和内部元数据。
6. 记忆主动回访只依据用户明确确认的 welcome / later / pause，不用浏览行为推断心理；偏好不隐藏、删除或降低馆藏与搜索可见性。
7. 不确定时间线只聚合四类受控时间来源；用户在一个日期、一个范围、保留多种记录、仍不确定之间明确选择，系统不回写原文或展品日期，也不裁决唯一真相。
8. 时间来源由内容稳定键与来源集合 SHA-256 锚定；来源变化只派生 `needsReview` 并保留旧判断，保存与删除使用 ETag / If-Match 防止静默覆盖。
9. 文字馆藏、媒体、声音、实体、展览、回访、回访意愿、时间校准、胶囊与记忆考古可由同一份自校验 `.time-isle` 归档迁移，同时保持 Vanilla JS + Node.js + SQLite 的小型可审计架构。
10. PWA 只提供可安装入口和诚实的断网边界页，不缓存主应用、API、图片、声音、归档或任何用户内容，也不冒充离线馆藏。
11. 展品有效变化进入带 SHA-256 父链的规范快照年轮；no-op 不制造版本，旧版恢复只新增 head，不覆盖历史，并以版本条件阻止并发丢失更新。
12. 馆藏体检、备份验真和时间线读取保持可解释、可取消或只读；严格脱敏、旧格式兼容、future schema 与崩溃 staging 清扫都在任何业务写入前形成边界，V2 离线载荷仍执行精确合同和会话清理。
13. 口述史只为符合条件的已确认事件确定性生成一个问题；录音/本地音频、人工选段、手填文字稿与 `day / range / uncertain` 都由用户明确完成，不自动转写、识别说话人或判断情绪。
14. 回答使用 `draft / confirmed / superseded / withdrawn` 追加状态；只有 `confirmed + day/range` 生成事件级 `oral-history` 候选，任何变化只让旧时间校准进入 `needsReview`，不回写原日期。
15. Schema 13 以 ETag/If-Match、`questionSetSha256`、稳定 `submissionId`、声音引用保护和严格 `.time-isle` 映射恢复维持并发、幂等与归档闭环；非空 JSON 只验真并返回 `requiresTimeIsle`。
16. V10 策展执行器固定为本地规则与四项只读工具，预算限制步骤、工具、时间、结果字节和来源数；不访问网络、文件、任意 SQL 或外部模型。
17. 执行只生成绑定请求/来源哈希的 `proposal / preview`；保存草稿、确认候选关系和发布拆成逐项 `confirm + If-Match + Idempotency-Key` 决定，分享保持独立最小披露流程。
18. 保存的步骤回执可离线重放评测，运行可取消；进程重启将遗留运行标为 `interrupted`，来源变化将旧提案标为 `needsReview`，二者都不自动续跑或执行副作用。
19. Schema 14 将运行、步骤、提案和决定纳入完整/脱敏归档；恢复重写 ID、隔离幂等域并强制为不可再次决定的历史，健康投影只公开安全计数。
20. 共忆信笺的请求与回信由浏览器端 PBKDF2-SHA-256 + AES-256-GCM 加密并绑定同一请求摘要；文件不自动发送，回信不自动入馆或合并原记忆。
21. 回信身份固定为自述未验证、文件固定为未签名；确认入馆必须重新校验请求/回信/来源合同，并以不可变独立来源进入归档与 provenance。
22. 馆外共忆文件保持密文，但确认入馆后的问答是本机 SQLite 普通明文；产品不能把文件传输加密扩大为数据库或磁盘静态加密承诺。
23. 设备内镜片按明确 ID 在服务端重读 2–20 件展品，只用确定性规则生成时间/共同出现/证据/线索投影，固定零外部模型、零工具调用、零保存。
24. 策展简报只接受用户明确选择的 2–6 件镜片来源；7–20 件必须重选，不能静默截断、自动运行、保存或发布。
25. Schema 19 用单例锁状态、revision CAS、不可变 verifier 和校验和提供 fail-closed 写门禁；锁馆等待在途写入与维护结束，后续 mutation 在读取正文前返回 423。
26. 锁馆只阻止经应用发起的新写请求，GET、导出、归档验真和结构演练仍可用；verifier 不进入日志和普通归档，但锁馆不加密 SQLite、媒体或磁盘。
27. 结构恢复演练复用完整归档预检并有界复核 manifest、哈希与引用，固定不执行当前馆藏恢复或隔离恢复，也不证明灾难恢复与磁盘加密。

当前发布版为 `14.0.0`、数据库 Schema `19`，要求 Node.js 24 或更高版本。2026-07-20，功能提交 `2dcce402b13f1d43c54c6a196b8e2273c9483eb3` 已进入发布链路，GitHub 与 Gitee 的 `main` 已对齐，Vercel 生产接口已核验为 `14.0.0 / schema 19 / interview-demo / ephemeral-sqlite / mock-fallback`。发布代码已通过完整 build/check、262 条真实 HTTP smoke、15/15 项 1265×720、390×844、320×700 Playwright 门禁和同三档真实目视。生产策展 sample 为 `synthetic + demo` 且调用前后持久化 runs 保持 0；锁馆与结构演练使用虚构 `text/plain` 正文的安全探针均返回 `403 / MUSEUM_LOCK_DEMO_READ_ONLY / bodyBytesRead: 0`，前后 stats、锁状态和 runs 不变。V10 的 249 条 smoke 及其具体专项数字只作为 2026-07-19 的历史发布基线，不代表 V14 的测试规模。

本地运行由 `server.js` 启动 Node HTTP Server；Vercel 的 `api/index.js` 导出同一个 request handler，避免两套路由漂移。媒体存储驱动当前只支持本地文件系统。

V7.1 的 `manifest.webmanifest` 继续以现有四主视图为唯一应用入口。根作用域 `sw.js` 只预缓存独立 `offline.html`、对应样式和公开品牌图标；在线导航始终优先访问服务，失败才返回边界页。API、主应用脚本、图片、声音、归档、非 GET 和跨源请求全部旁路，Worker 也不调用 `cache.put` 动态收集响应。更新不会 `skipWaiting`、不会强制刷新或抢占旧页面，只删除 `time-isle-public-shell-` 自身前缀的旧缓存。所有图片变体与声音响应统一使用 `private, no-store`，安装不会把私人馆藏搬进浏览器 HTTP 缓存或 Cache Storage。

HTTP 入口在解析 URL 与等待维护任务前先通过 `lib/request-security.js`：本地模式只允许三种规范回环 Host；Vercel 模式只允许平台注入域名和 `ALLOWED_HOSTS` 中的精确主机。写请求必须带精确同源 Origin，并拒绝浏览器标记为 `same-site` / `cross-site` / `none` 的 Fetch Metadata。这条边界阻止 DNS rebinding 使互联网页面借用用户本机服务进行读写。

## 模块边界

- `lib/media-format.js`：JPEG、PNG、WebP 的真实容器、尺寸、静态派生图和像素限制校验。
- `lib/media-storage.js`：流式暂存、派生图接收、内容寻址落盘、隔离删除和过期暂存清理。
- `lib/demo-safety.js`：把 Demo 的启动清理限制在系统临时目录内的项目专用路径，并解析符号链接 / 目录联接后的真实边界。
- `lib/request-security.js`：规范化单一 Host，区分本地 / 部署白名单，并校验写请求的 Origin 与 `Sec-Fetch-Site`。
- `lib/media-database.js`：媒体资产、变体、展品关联、observations 与引用约束。
- `lib/media-api.js`：HTTP 路由、EXIF 线索落库、相似候选、ETag 和孤儿回收协调。
- `lib/media-evidence.js`：规范化图片区域、来源哈希锚点和公开证据形态。
- `lib/exif-hints.js`：不可信 JPEG APP1/Exif 的有界、异常安全解析。
- `lib/media-similarity.js`：确定性 `dhash64-v1` 与只读候选分类。
- `lib/time-isle-archive.js`：无依赖的确定性 ustar + gzip 创建和严格解包。
- `lib/media-backup.js`：归档清单、媒体选择、哈希/MIME/引用全量验证。
- `lib/media-restore.js`：ID 映射、内容复用、文件物化、SQLite 事务与失败补偿。
- `lib/archaeology.js` / `lib/archaeology-backup.js`：文字证据算法和事件关系迁移。
- `lib/exhibition-curator.js` / `lib/exhibition-database.js` / `lib/exhibition-api.js`：引用约束下的主题策展、发布、待复核和恢复映射。
- `lib/revisit-service.js` / `lib/revisit-database.js` / `lib/revisit-api.js`：本地日期与时区语义、稳定候选、当日去重、明确意愿筛选/排序和 Demo 零写入。
- `lib/revisit-intent-database.js` / `public/assets/revisit-intents.js`：Schema 11 约束表、确认式 welcome/later/pause、自然回访清除、按需管理与完整/脱敏备份合同。
- `lib/time-calibration-service.js`：规范化日/范围/多种记录/不确定判断，从当前展品、修订、有效原文 claim 与非敏感 EXIF 生成候选，并为单个来源和来源集合构造稳定 SHA-256 锚点。
- `lib/time-calibration-database.js`：Schema 12 约束表、memory/event 互斥目标、事务保存、时间线投影、`needsReview` 派生及完整/脱敏备份恢复。
- `lib/time-calibration-api.js` / `public/assets/time-calibrations.js`：GET/PUT/DELETE、ETag / If-Match、Demo 403、默认折叠来源校准台、会话中止与旧响应隔离。
- `lib/oral-history-service.js`：确定性问题资格、来源快照、人工选段/文字稿/时间含义归一化，以及事件级 `oral-history` 时间候选。
- `lib/oral-history-database.js`：Schema 13 两表、追加式回答状态、supersede/withdraw、幂等提交、声音使用量与完整/脱敏备份恢复。
- `lib/oral-history-api.js` / `public/assets/oral-histories.js`：事件 GET/PUT/DELETE、ETag/If-Match、`questionSetSha256`、`submissionId`、Demo 403 与默认折叠的多步录入。
- `public/assets/voice-capture.js`：单段 MediaRecorder、本地 WebM/M4A、取消/重试、对象 URL 回收与人工选段所需元数据；公开 Demo 不启动采集。
- `lib/oral-history-backup.js`：schema 13 `oral-history/state.json` 的精确完整/脱敏合同、事件/声音边界和恢复前交叉校验。
- `lib/curator-agent-service.js`：固定预算、四项只读工具、确定性本地提案构造、来源摘要与无副作用重放评测。
- `lib/curator-agent-database.js`：Schema 14 四表、运行/取消/中断状态、ETag 版本、逐项幂等决定和只读历史恢复。
- `lib/curator-agent-api.js`：运行列表/创建/执行/取消/评测/决定/删除路由，以及 Demo 的合成 sample 与统一非只读 403。

- `public/assets/co-memory-crypto.js` / `public/assets/co-memory-letter.js`：一问一答请求与回信的 Web Crypto 协议、自包含邀请页、请求摘要绑定、错口令/篡改拒绝和会话清理。

- `public/assets/memory-lens.js` / `public/assets/memory-lens-host.js`：默认折叠交互、2–20 件选择、2–6 件内存简报交接，以及 7–20 件明确重选而非截断。

## 数据模型

### 文字与记忆考古

原有核心表继续保留：

- `memories`：原文、展品说明、展厅、日期、地点、重要度和兼容字段。
- `memory_people`、`memory_tags`、`memory_emotions`：展品多值字段。
- `agent_runs`、`agent_steps`、`agent_events`：一次整理的三阶段处理轨迹。
- `memory_events`、`event_members`：用户确认后的同一往事及多个独立版本。
- `memory_claims`：字段、原文摘录、UTF-16 字符位置与来源状态。
- `memory_pair_decisions`：规范化 pair key 对应的人工决定。
- `curator_questions`：补充问题、回答、跳过和保留不确定。

### 三张媒体表 + observations

`media_assets` 是逻辑资产表：

- 以 `content_sha256` 唯一约束实现精确内容复用。
- 记录源 MIME、源字节数、规范展示尺寸、隐私模式、存储驱动和生命周期状态。
- 状态为 `staging / ready / pending_delete`，当前驱动固定为 `local`。
- `safe_metadata_json` 记录规范变体、坐标空间和源尺寸等安全元数据。

`media_variants` 是物理文件目录：

- 一个资产可有 `original / display / thumb` 三种变体。
- 每个变体独立记录存储键、MIME、字节数、尺寸和 SHA-256。
- `(asset_id, kind)` 为主键；存储键全局唯一。
- `sanitized_only` 不创建 `original` 变体。

`memory_media` 是展品与资产的多对多关联：

- 主键为 `(memory_id, asset_id)`，删除展品会级联解除关联，删除仍被引用的资产会被数据库阻止。
- 记录 `cover / gallery / evidence` 角色、顺序、说明、替代文字、照片背面与关联元数据。
- 部分唯一索引保证每件展品最多一张封面。

`media_observations` 是统一线索账本：

- `kind/source/value/status/confidence/sensitive/metadata` 将线索来源和确认状态分开。
- `status` 为 `suggested / confirmed / rejected`；来源支持 `user / exif / ocr / system / model / import`。
- 当前用于 EXIF 建议、检索指纹和用户确认的 `image_region`。OCR 草稿只有在用户确认后才通过图片区域 API 保存，最终语义来源仍是用户确认，而不是机器事实。
- observation 随资产删除而级联清理；与展品相关的区域在 metadata 中保留 memory ID。

`database.js` 的 `withTransaction()` 支持同步嵌套调用，使文字导入、媒体创建、关联、observations 与考古恢复可以共享外层原子事务。

### 展览、回访、实体与声音

Schema 5–8 通过带校验和的 `schema_migrations` 账本顺序增加四组能力：

- `exhibitions / exhibition_sections / exhibition_items / exhibition_citations`：展览只有在用户确认、引用仍可回到原文且无需复核时才能发布。来源展品被编辑或删除后，相关引用会重新核验，展览降级为草稿或被收拢。
- `memory_revisit_state`：保存最近查看日期、次数与当日本地隐藏状态；候选依赖调用方明确提交的 IANA 时区和本地日期，同一天已打开或略过的记忆不会重复出现。
- `entities / entity_aliases / memory_entities / memory_search_documents`：把人物、地点、标签和确认声音文字稿写入可解释检索文档。同名默认仍是不同身份；别名新增和实体合并必须先预览，再明确确认。
- `voice_assets / memory_voice / voice_transcripts`：声音文件与展品多对多关联，文字稿具有 `draft / confirmed` 状态。只有人工确认稿进入检索、详情展示、胶囊和离线分享；原始音频首版不进入分享文件。

### 时间胶囊三表

Schema 9 新增：

- `time_capsules`：只保存标题、外壳留言、`opens_on`、IANA 时区、来源展览引用、复核状态与时间戳。
- `time_capsule_payloads`：单独保存匿名安全快照 JSON 与对精确 UTF-8 JSON 计算的 SHA-256；外壳列表和锁定读取路径不查询这张表。
- `time_capsule_media`：只引用用户封存时勾选的 `display` WebP，保留匿名 `item_key`、顺序、替代文字和说明；图片资产删除会级联移除引用，并把胶囊标记为待复核。

胶囊创建只接受 `status: published`、`needsReview: false`、所有引用仍有效的主题展览。快照会重新生成匿名章节与 `item-1` 等规范键，只保留有效 `confirmedQuotes`、用户选中的确认文字稿及安全展示图引用；展品 ID、展览 ID、图片/声音内部 ID、原图 URL/哈希、EXIF/GPS、Agent 轨迹和草稿文字稿不会跨越快照边界。

### Schema 10：记忆年轮

Schema 10 新增 `memory_revisions`，以 `(memory_id, revision_no)` 保证每件展品从 1 开始连续编号。每条记录保存规范快照 JSON、快照 SHA-256、父快照 SHA-256、修订类型、可选备注、来源更新时间与创建时间；`restored` 还必须引用同一展品链中更早的修订 ID。修订随展品删除级联清理，不保存 Agent run、媒体二进制、声音二进制或其他展品的引用图。

### Schema 11：长期回访意愿

Schema 11 新增 `memory_revisit_intents`，以 `memory_id` 为主键并外键级联到展品。`intent` 只允许 `welcome / later / pause`；表级 CHECK 强制 `later` 同时携带非空 `not_before_local_date / not_before_timezone`，其余两种必须同时为空。索引按意愿、延期日期、更新时间和展品键排序；单库最多保存 500 条，重复保存相同语义保持 no-op。`neutral` 不落表，恢复自然回访就是删除该记录。

### Schema 12：不确定时间线校准

Schema 12 新增 `time_calibrations`。每条记录必须且只能引用一个 `memory_id` 或 `event_id`，两个目标都带外键级联和唯一约束，因此每件展品或每个已确认事件最多有一份当前判断。`resolution_kind` 在数据库层允许 `year / month / day / range / alternatives / uncertain`；V8 界面只开放“一个日期（day）/一个范围（range）/保留多种记录（alternatives）/仍不确定（uncertain）”四个清晰选择，`year / month` 只用于规范化历史或结构化来源兼容。

`alternatives / uncertain` 强制空区间，其余类型强制非空规范区间；`source_set_sha256` 必须是 64 位小写 SHA-256，所选来源以去重、排序后的稳定 `time-source:<sha256>` 键数组保存，备注最多 500 字符。保存记录还携带所选来源的最小不可变快照，精确字段只有 `sourceKey / sourceType / precision / intervalStart / intervalEnd`，不复制展品标题、摘录、整段原文、媒体 URL 或 GPS。单库最多 500 条校准，目标和来源键长度也有硬上限。时间校准与展品日期、修订快照、claim、EXIF observation 分表保存，因此更新或删除判断不会改动任何来源对象。

### Schema 13：事件级口述史

Schema 13 通过独立迁移新增 `oral_history_questions` 与 `oral_history_answers`，不把表创建塞进旧声音迁移，避免迁移账本越级。问题表按事件保存稳定问题键、问题文本、原始来源快照及来源集合摘要；回答表引用问题与 `voice_assets`，保存 `segment_start_ms / segment_end_ms`、人工文字稿、`resolution_kind`、可选闭区间、稳定 `submission_id` 和追加式状态时间。问题与回答都有数量、长度、ID、时间和外键约束。

回答状态限定为 `draft / confirmed / superseded / withdrawn`。确认新回答时，事务先验证事件、问题集合、声音 ready 状态、选段在音频时长内、文字稿与时间语义，再把旧 current 回答标记为 `superseded` 并插入新记录；撤回只改变当前状态，不物理删除历史。`submission_id` 在同一事件范围内稳定去重，配合 API ETag 支持网络重试幂等重放。

只有 `confirmed` 且 `resolution_kind` 为 `day / range` 的当前回答进入时间候选，来源类型为 `oral-history`，身份锚定事件级问题而非某件成员展品；`uncertain` 与 `draft` 不生成结构化日期。候选新增、替换或消失会改变时间来源集合摘要，从而让旧 `time_calibrations` 派生 `needsReview`，但数据库不会更新展品日期、修订、claim、EXIF 或既有校准内容。

### Schema 14：受限策展审计

Schema 14 新增四张相互引用的表：

- `curator_agent_runs` 保存精确请求 JSON、请求摘要、固定预算、用量、`created / running / completed / cancelled / failed / interrupted` 状态、并发版本、幂等域、历史/待复核/可决定标记和终态时间。
- `curator_agent_steps` 只接受四项只读工具，按 `(run_id, position)` 唯一并连续保存参数、结果、字节数、耗时、摘要和结果 SHA-256；外键随运行级联，触发器禁止更新回执。
- `curator_agent_proposals` 每个运行最多一条，保存本地规则版本、请求/来源集合/提案摘要、2–6 个来源快照、展览预览、可选关系、三项动作声明与重复展览上下文；触发器禁止更新。
- `curator_agent_decisions` 以 `(run_id, action)` 和全局 `idempotency_key` 双重唯一，保存 `approve / reject`、请求摘要和最小副作用回执；触发器禁止更新。

运行表的 CHECK 将预算固定为 6 步、4 次工具、2,000 ms、262,144 结果字节和 6 件来源，同时限制步骤/工具计数、JSON 类型、SHA-256、历史授权和终态组合。数据库启动会把遗留 `running` 原子标记为 `interrupted` 并关闭决定权限；中断运行不能恢复执行，只能保留审计或创建新运行。

### Schema 15–18：来源收件、护照与共忆见证

- Schema 15 的 `memory_inbox_sources / memory_inbox_items` 保存源文件摘要、规范文本身份、逐字片段锚点与人工接纳状态；整份 TXT/Markdown 不落库，片段和新展品在同一事务确认。
- Schema 16 的 `provenance_claims / provenance_claim_sources / provenance_claim_events` 把来源护照建模为不可变主张、冻结来源集合和追加事件链；来源变化只触发待复核，不改写原记忆。
- Schema 17 的 `co_memory_responses` 保存经显式确认、与请求摘要绑定的独立共忆回信；记录固定 `identity_verified = false`、`encrypted = true`、`signed = false`，并由触发器禁止原地修改。
- Schema 18 重建 `provenance_claim_sources` 的枚举约束，使 `co_memory_response` 能作为独立来源进入来源护照；迁移必须连续完成，不能靠 `INSERT OR REPLACE` 绕过冻结来源约束。

### Schema 19：锁馆单例与 fail-closed 状态

`museum_lock_state` 只允许一行固定身份记录，保存 `locked / unlocked`、revision、派生 verifier、转换回执与校验和。触发器禁止删除、替换、身份变更、revision 跳跃和 verifier 更换；写入必须使用 compare-and-set 更新。应用读取时还会重算公开状态投影与校验和，缺表、缺行、非法枚举或内容损坏都按锁定处理，而不是默认回到可写。

verifier 由口令和随机 salt 经受预算约束的 scrypt/PBKDF2-SHA-256 派生；明文口令不进入 service 或 database 状态。API 只公开 `status / revision / verifierConfigured`，不返回 salt、digest 或 KDF 细节。普通 JSON、`.time-isle` 和日志也明确排除整个锁状态，避免把认证材料当成馆藏数据迁移。

## 记忆年轮与并发写边界

快照先把标题、原文、展签、日期、地点、展厅、人物、标签、情绪、附件等允许字段规范化，再对精确 UTF-8 JSON 计算 SHA-256。首条修订只允许 `baseline / created / imported` 且父哈希为空；后续只允许 `edited / restored`，其 `parentSha256` 必须等于前一版 `snapshotSha256`。序号断层、哈希不匹配、非规范集合顺序、额外字段和非法恢复引用都会被拒绝。这条链用于发现内容或顺序被破坏，不是外部签名、可信时间戳或不可篡改审计账本。

展品 `PUT` 与旧版恢复都使用当前 `updatedAt` 派生的 ETag。浏览器发送 `If-Match`；缺少版本条件返回 `428`，条件过期返回 `412` 和新的版本提示，不把当前私人正文塞进冲突响应。兼容调用方仍可在请求体提交 `expectedUpdatedAt`，数据库层执行相同的比较后写入。内容快照没有变化时保持 no-op：不新增修订、不更新时间、不改变 ETag。

恢复操作先按需读取旧快照，再经二次确认把它复制成新的最新版本，类型为 `restored` 并记录 `restoredFromRevisionId`；当前版本和中间历史都不会删除。修订写入与展品更新共享同一 SQLite 事务，任一步失败都不会留下孤立 head。

## 明确回访意愿边界

`GET /api/revisits/:memoryId/intent` 读取单件公开形态，`PUT` 只接受 `choice / notBeforeLocalDate / timezone / confirm` 精确键集且要求 `confirm: true`；`GET /api/revisits/intents` 返回已设置意愿及最小展品摘要。Demo 模式的所有意愿写入与清除返回 403。

候选选择仍先由“往年今日、很久没见、随机漫游”产生，再由 `applyRevisitIntents()` 做确定性处理：

- `pause` 直接从主动回访候选中过滤。
- `later` 使用保存的 IANA 时区把当前时刻换算为本地日历日期；到期前过滤，到期后恢复可用并在依据中显示用户选择的日期。
- `welcome` 不绕过原回访类型条件，只把合格候选稳定提升到前面。
- 只有 `welcome / later` 会在返回依据中加入 `source: user-confirmed`；没有意愿时维持原顺序。

浏览、略过、搜索与情绪字段都不会自动写入意愿，数据库也不保存选择原因。意愿不改变展品、FTS、详情或归档读取权限；它只控制系统何时主动带回一件记忆。前端单件设置和全局管理均默认折叠、按需请求，并在会话切换时中止旧请求。

## 不确定时间线与来源校准边界

### 四类受控来源

`buildTimeCandidates()` 只把以下来源转换为规范时间候选，其他字段、模型自由文本和 GPS 都不能进入来源集合：

1. `memory-current`：展品当前结构化日期及最小标题/ID；它表示“当前记录是什么”，不表示优先级或真实性。
2. `revision`：记忆年轮快照中的日期，必须同时有合法 `snapshotSha256`，并可附修订序号。
3. `raw-claim`：类型为日期、`evidenceValid: true` 且状态未失效/拒绝/撤回的原文 claim；摘录、起止字符区间和摘录 SHA-256 必须互相一致。
4. `exif`：关联照片中 `source: exif`、`kind: captured_at`、非敏感且状态为 `suggested / confirmed` 的日期；候选锚定媒体内容 SHA-256，可记录时区类型，但绝不读取或输出 GPS。

年、月、日与范围先规范为包含起止日的闭区间。每个候选只把来源类型/状态、精度、区间和对应内容锚点纳入规范 JSON，再生成稳定 `sourceKey`；展示标题等易变文字不参与键计算。对去重并排序后的全部来源键再次计算 `sourceSetSha256`。这使“来源内容发生变化”和“界面文案变化”可以被区分，也避免把数据库自增 ID 当作跨归档依据。

### 四类人工选择与 `needsReview`

前端只提交 `day / range / alternatives / uncertain`：精确日把起止日设为同一天，范围要求开始日不晚于结束日，后两者必须为空区间。选择 `day / range` 时，保存区间必须与每个已选择来源区间相交；选择 `alternatives` 时，所选来源中必须至少存在一对互不相交区间，否则返回语义校验错误；`uncertain` 可以诚实表达当前无法收敛。保存只产生独立 `time_calibrations` 记录和最小来源快照，不调用展品更新、修订恢复、claim 写入或 EXIF 确认路径。

读取 workspace 时重新生成当前候选集合。只要已保存 `sourceSetSha256` 与当前集合摘要不同，响应派生 `needsReview: true`；数据库中的旧判断、原所选来源键、最小不可变来源快照与备注仍完整保留。来源消失不会让已保存判断变得不可解释，时间线也不会把待复核解释为“旧日期错误”。用户可刷新来源后更新判断，或明确删除校准以恢复未校准状态，两种操作都不回写原文与展品日期。

事件级校准用于人工确认的同一往事版本组，并在时间线中优先于成员级判断；成员级记录不会被删除。`GET /api/timeline?order=asc|desc&limit=1..500` 使用独立公共投影，严格只返回最小目标摘要、可展示的判断摘要、`needsReview` 和最多两个确认成员 `memberIds`，让界面可以重新打开对应拼图；原文、媒体、备注、所选来源键、不可变来源快照与来源集合 digest 均不跨越该接口。只有“已确认事件 + 两侧存在可验证日期差异”时，时光拼图才显示默认折叠的校准台并接管日期补问；讲解与回顾只放默认折叠的“不确定时间线”摘要，四项主导航不变。

### HTTP 并发与 Demo 只读

`GET /api/time-calibrations/memories/:id` 与 `GET /api/time-calibrations/events/:id` 返回目标、候选、已保存判断、当前来源集合摘要、`needsReview` 和 ETag。`PUT` 请求体必须精确包含 `confirm / resolutionKind / intervalStart / intervalEnd / selectedSourceKeys / sourceSetSha256 / note`，`DELETE` 必须精确包含 `confirm`；两者都要求 `confirm: true` 和当前 `If-Match`。缺少前置条件返回 `428 CALIBRATION_PRECONDITION_REQUIRED`，ETag 过期返回 `412 CALIBRATION_VERSION_CONFLICT`，冲突响应不携带新的私人正文。

详情接口最多下发 100 条候选，并显式返回 `candidateCount / candidatesTruncated`；服务端确定性优先保留已选来源、当前记录、原文 claim 与 EXIF，再补历史修订，避免前端按哈希顺序静默截断。仍可用的已选来源保证进入这 100 条，已经失效的来源则由只读最小快照解释。

ETag 覆盖目标、判断语义、来源集合摘要与更新时间；空 workspace 也有稳定 ETag，因此首次创建同样不能绕过并发条件。前端为每次打开/刷新建立新的 `AbortController` 与 session key，旧请求即使忽略 abort 也不能把来源或备注写回新拼图。`INTERVIEW_DEMO=true` 时 GET 和预置 `alternatives` 示例保持可读，任何校准 PUT/DELETE 在业务入口统一返回 `403 CALIBRATION_DEMO_READ_ONLY`。

## 一问一答口述史边界

### 问题资格与人工录入

`buildOralHistoryQuestionSet()` 只处理状态为 confirmed 的事件，并要求双侧各有可验证日期来源且至少一对区间互不相交；已经由确定校准解决、来源单侧缺失或没有真实冲突时不生成问题。问题文本与来源集合都由确定性规则产生，不调用模型。前端继续位于 `puzzleBody`，`oralHistoryDetails` 默认折叠，与 `timeCalibrationDetails` 并列且不增加导航。

录入顺序固定为选择问题、录音或本地音频、设置起止毫秒、手填文字稿、明确确认文字稿与 `day / range / uncertain`、保存。`voice-capture.js` 只负责本次单段采集；用户可取消、重试或移除音频，临时对象 URL 在替换/关闭时回收。两个原生 range 支持“以当前播放位置设为开始/结束”和只试听选段。服务端重新验证真实声音格式、ready 资产、总时长与片段边界，不能信任浏览器声明。

### 回答状态、并发与时间来源

`GET /api/oral-histories/events/:id` 返回最小事件摘要、资格、当前问题、`questionSetSha256`、当前 draft/confirmed、最多 100 条折叠历史与 ETag。`PUT` 保存回答，`DELETE` 撤回当前回答；两者都要求当前 `If-Match`。PUT 还要求问题身份、问题集合摘要、音频资产、片段、人工文字稿、明确确认、时间含义与稳定 `submissionId`。过期 ETag 返回 `412`；相同 `submissionId` 的已完成请求可以在 ETag 已变化时幂等返回，不重复插入回答。

`confirmed + day/range` 由时间校准层读取为事件级 `oral-history` 来源；`confirmed + uncertain` 只进入口述证据卡，`draft` 不进入候选。新确认回答 supersede 旧回答，撤回使用 `withdrawn`；旧声音片段与文字稿默认折叠但仍可审计。任何回答变化只改变当前来源集合并派生 `needsReview`，不会调用时间校准 PUT，也不会写回原记忆。

声音资产的显式删除、解除关联与 GC 都调用统一使用量函数；口述史回答引用计入额外 usage。只要回答仍引用资产，即使它没有 `memory_voice` 关联也不能被当作孤儿删除。

### Demo 与隐私

`INTERVIEW_DEMO=true` 允许 GET 只读展示符合条件的问题与已有来源，但 PUT/DELETE 固定返回 403。前端在 Demo 状态不调用 `getUserMedia`、不触发文件选择、不上传音频，也不发送口述史写请求；没有真实示例音频时只展示禁写问题入口，不伪造可播放回答。系统不自动转写、识别说话人、分析情绪或把声音送往外部模型，也不自动发布、分享或挂靠某件成员展品。

## 受限本地策展助手边界

### 固定预算与只读工具

`executeCuratorAgent()` 不接受动态工具注册。构造时必须提供且只能执行四个名称：`search_memory_summaries`、`read_memory_evidence`、`read_confirmed_relationships`、`read_exhibition_summaries`。工具输入和结果各有精确键集、列表和文本上限；所有结果先规范化，再记录 UTF-8 字节数与 SHA-256。执行器不导入 AI 客户端，不读环境 Key，也不提供网络、文件、任意 SQL、写数据库或运行 shell 的能力。

固定预算 `maxSteps: 6 / maxToolCalls: 4 / maxDurationMs: 2000 / maxResultBytes: 262144 / maxMemories: 6` 同时由服务、数据库 CHECK、归档 validator、API 投影与界面回执约束。运行建立后必须显式 `confirm: true` 执行；取消和进程重启会终止运行语义，任何 `interrupted` 记录都不能续跑。执行只保存四项只读回执并构造一份 `curator-agent-proposal`，不触发副作用。

### 来源绑定、重放评测与待复核

请求经稳定 JSON 生成 `requestSha256`；每个来源只保存 memory ID、原文 SHA-256 和更新时间，排序后生成 `sourceSetSha256`；提案的预览、候选关系、动作声明、重复上下文和来源再生成 `proposalSha256`。`evaluateCuratorAgentTrace()` 只读取保存的运行、步骤和提案，重新核对工具顺序、预算、结果字节/哈希、来源闭包与提案构造，不重新调用工具，也不重放任何决定副作用。

用户决定前，数据库重新读取当前来源并比较原文摘要和更新时间；任一来源变化会把运行标为 `needsReview`、关闭 `allowDecisions` 并返回冲突。旧提案和回执仍可只读回看，用户必须新建运行，不允许系统用变化后的来源静默改写旧提案。

### 逐项决定与 HTTP 并发

API 将副作用限制为三个具名 action：

- `save_exhibition` 只创建 draft 展览。
- `confirm_relationship` 只接受提案中确实存在的候选关系。
- `publish_exhibition` 依赖先前 `save_exhibition` 成功回执中的展览 ID，只把该草稿发布到本馆。

每个 action 最多有一条最终决定。请求体必须精确包含 `action / decision / confirm`，且 `confirm` 为 true；HTTP 同时要求当前运行 ETag 和 `Idempotency-Key`。完全相同的重试返回既有回执，不同请求复用幂等键返回 409，过期版本返回 412。批准与数据库决定回执处于同一事务边界；拒绝只记录无副作用结果。分享没有 action，发布成功后前端仅提供前往既有隐私编辑台的入口。

### Demo 合成 sample

`GET /api/curator-agent/sample` 通过纯函数生成固定运行、四项回执、提案和 evaluation；它不调用 store，也不创建 SQLite 记录。`INTERVIEW_DEMO=true` 时，策展 API 在路由分派、请求体读取、ETag 和幂等解析之前统一拒绝全部非 GET/HEAD 请求。运行列表固定为空、任意具体运行返回 404；页面只展示 sample，并明确“未保存、不可决定”。

## 共忆信笺的加密、确认与来源边界

### 馆外一问一答协议

`co-memory-crypto.js` 为 request 与 response 使用不同格式标识和 AAD 绑定。请求包含一个问题、最小上下文与内嵌 memory anchor；规范请求计算 SHA-256 后同时进入信封 binding 与回信 payload。两类信封都使用随机 salt/IV、PBKDF2-SHA-256（310,000 次）和 AES-256-GCM，解密后重新验证精确键集、版本、请求摘要和 letter ID。篡改、错口令、跨邀请回信和非规范 Base64 都在进入确认流程前拒绝。

邀请 HTML 内联必要脚本与样式，CSP 不允许网络连接；对方填写的身份标签只是一段自述文字，协议强制 `assurance: self-asserted-unverified / verified: false`。回复导出为加密 JSON，不自动发送。文件没有数字签名、账号认证、可信时间戳、撤回或口令找回能力。

### 入馆确认与明文存储边界

馆主侧先在浏览器解密请求和回信，再将规范化 plaintext contract 交给 `/api/co-memory-responses/confirm`。服务端不信任客户端“已核对”的声明：它重新计算请求摘要、核对 letter/response 绑定、memory anchor、来源投影、`identityVerified: false / encrypted: true / signed: false`，并要求 `confirm: true`、稳定幂等键和目标记忆存在。

保存后，问题、回答、身份标签与来源锚点作为不可变记录进入 SQLite；这一步已经离开馆外加密信封。换言之，PBKDF2/AES-GCM 保护的是交换文件，不是数据库静态加密。普通本地查询、管理员或直接读取数据库文件仍可能看到明文，必须由系统账号和设备磁盘加密补足。

完整归档先恢复共忆回信，再恢复引用它的 provenance source；任何目标 ID、请求/回答哈希、anchor 或来源集合冲突都使同一事务回滚。脱敏归档严格只留下响应数、未验证身份数、加密传输数与未签名数，排除问答、身份标签、内部 ID、source key、anchor 与哈希。

## 设备内可解释镜片边界

`GET /api/memory-lens/preview` 只接受一次 `lens`、2–20 个不重复 `memoryId`，以及仅在线索镜片出现的 `query`。未知参数、重复 ID、缺失展品和非 GET 方法都拒绝。服务端依次 `store.getMemory()` 并补充当前确认实体/文字稿，再将保存态对象传给纯函数；客户端不能上传或替换正文。

四种投影均输出排序/分组原因、证据字段、来源引用、`sourceSnapshotSha256 / requestSha256 / previewSha256` 与固定边界：

- 时间镜片只按明确保存的年/月/日排序；非规范日期保持文字，空日期保持未知，不从标题或正文补全。
- 共同出现镜片只读取已确认实体引用；共现不表示朋友、亲属或其他关系。
- 证据镜片只统计原文、确认引用、图片和确认文字稿；数量不表示真实性或质量。
- 线索镜片只对用户给出的 1–8 个明确词做直接字段匹配；不扩展近义词、不调用模型，也不把命中解释成事实。

执行回执固定 `deterministic: true / externalModel: false / modelCalls: 0 / toolCalls: 0 / persisted: false`。前端可把当前镜片变成一次内存简报，但 `preselectSources()` 只接受 2–6 件明确选择；7–20 件返回重选提示，绝不自动截断。交接结果不自动发起策展运行，也不保存或发布任何内容。

## 锁馆写门禁与结构恢复演练

### 写请求前置门禁与转换协调

`museum-write-runtime.js` 在 Content-Type、正文读取和业务路由之前分类请求。GET/HEAD/OPTIONS 不是 mutation；锁馆/解锁控制、`POST /api/archive/inspect` 和 `POST /api/recovery-drills/structural` 是明确只读例外。其他 mutation 在 `locked` 或 transition pending 时返回 `423`，决定对象固定 `bodyBytesRead: 0`。锁馆转换先设置 pending，拒绝后来到达的写请求，再等待 `activeMutations` 与后台维护归零；锁定时定时媒体维护直接跳过。

锁馆/解锁要求精确 confirmation、当前 revision、operation ID 与口令校验。相同 operation ID/输入可幂等重放，不同输入复用则冲突；错口令和过期 revision 不改变状态。锁状态读取缺表、缺行、枚举/校验和损坏时 fail-closed，不能因为数据库异常悄悄恢复可写。

口令只在一次浏览器请求内存中存在；关闭折叠层、请求结束或离页都会清空。服务端用既有 salt/KDF 重新派生 verifier并做常量时间比较，数据库从不接收明文口令。API 公共投影不含 `recoveryVerifier / salt / digest`，普通馆藏导出、`.time-isle`、日志和结构演练也不包含这些认证材料。

锁馆不是静态加密：它不会修改或加密 SQLite、FTS、图片、声音、staging 或备份文件，也不能约束绕过 Node 服务的本机文件访问。它只承诺“正常应用路径不接受新的写入”，不承诺冷态磁盘不可读。

### 只读结构验真而非恢复成功

`POST /api/recovery-drills/structural` 只接受完整 `.time-isle`。runner 只有 `prepareArchive` 依赖，没有 store、restore、import、database 或 media materialization callback；归档先进入隔离暂存和正式 preflight，再在固定预算内复核 manifest 条目、逐项哈希绑定、记忆/媒体引用与 observation 归属。当前馆藏和媒体写入计数固定为 0。

公共结果的 `kind` 固定为 `structural-verification`，并强制 `actualRestorePerformed=false`、`isolatedRestorePerformed=false`、`disasterRecoveryProven=false`、`diskEncryptionProvided=false`。因此页面只能说“结构验真通过”，不能说“恢复成功”。锁馆期间该接口保持可用；Demo 则在 query/header/body 之前返回 403，确保上传字节为 0。

## 时间胶囊开启与离线加密

### 日期只是仪式门槛

服务端使用胶囊记录的 IANA 时区把当前时刻换算为本地日期，并与 `opens_on` 比较。未到期或待复核时：

- `GET /api/capsules/:id/content` 返回 `423`，响应体严格只有 `error / code / capsule`。
- `capsule` 是公开外壳，不包含来源 ID、图片 ID、文字稿、快照摘要或 payload 大小。
- 处理路径不会调用 payload reader，因而不是“读出后再删字段”。

到期后才读取并校验 payload SHA-256，再把仍存在的安全展示图物化为公开内容 URL。这个日期门槛用于产品仪式感，不是不可破解的密码学时间锁；真正需要跨设备保密时必须生成加密离线文件。

### 浏览器端口令加密

服务器返回匿名安全快照并读取用户预选的 WebP 字节后，控制器先进入独立 `share-privacy.js` 三层编辑台，而不是直接显示口令：

1. 公开层使用通用默认标题“时屿加密分享”、固定说明和稳定文件名；不会复制来源展览/胶囊标题或导出时间，`opensAt` 固定为 `1970-01-01T00:00:00.000Z`，表示立即可打开。
2. 加密层保存受众、用途、可编辑叙事副本，以及章节、展品、引用、confirmed 文字稿和 display WebP。所有选择初始为 false；章节与展品必须逐级选择，取消父级会同步清除后代选择。
3. 收据层精确显示公开内容、密文内容数量、固定排除项和不可撤回边界。至少 1 章、1 件展品及 1 条引用或文字稿，并勾选最终确认后，口令区才出现。

`projectSharePayload()` 生成 V2：章节、展品和媒体连续重编号，移除未选证据与孤儿媒体，收据只含 `audience / purpose / counts / boundary`。结果不携带来源内部键、URL、SHA、未选原文、原图、EXIF/GPS、Agent 日志、草稿文字稿或原始音频。叙事编辑仅改变本次副本，不回写展览。

口令：

- 至少 12 个字符，只存在于当前浏览器内存；不会进入请求体、SQLite、日志、文件名、LocalStorage 或 SessionStorage。
- 使用 PBKDF2-HMAC-SHA-256，随机 16-byte salt，固定 310,000 次迭代，派生 AES-256-GCM 密钥。
- 每次导出使用随机 12-byte IV；`header + shell` 的规范 JSON 作为 AAD，密文自带 128-bit GCM 认证标签。
- 编辑已确认内容会立即撤销确认并清空口令。加密任务捕获同一份 prepared 对象；等待 PBKDF2/AES 期间若重新编辑或确认，新旧对象身份不一致会使旧任务以 stale 结束，不能下载修改前内容或混用新文件名。
- 成功或失败后都清空两次口令输入；取消、关闭、返回书架和下载后还会清除编辑草稿、明文媒体与被跟踪的 Blob URL。错误口令与文件篡改只得到统一的“无法开启”提示。

`capsule-crypto.js` 在加密前和解密后验证载荷。V2 要求精确字段、连续匿名键、非空章节/展品、至少一条文字证据、唯一媒体所有权、规范 Base64 与真实字节数、收据计数和固定边界完全一致；V1 保持原格式兼容并允许历史文本形态。主模块和内嵌阅读器统一按 Unicode 码点计数。

下载结果是自包含 HTML：CSP 为 `default-src 'none'`、`connect-src 'none'`，脚本和样式内联，图片从解密后的 WebP 字节创建短生命周期 Blob URL。公开外壳和 KDF 参数可见，受众、用途、展览叙事、证据和图片字节都在密文内；解密后先显示分享收据。单文件不含远程字体、统计脚本、网络请求、原始音频、原图或 EXIF。下载后应用没有账号撤回、远程失效或口令找回能力，知道口令的人仍可复制、转发或截图。

## 图片完整生命周期

### 1. 选择与轻量前检

前端从 `/api/options.mediaPolicy` 读取策略，默认限制包括：

- 仅 JPEG、PNG、WebP。
- 原图不超过 20 MiB、派生图不超过 4 MiB、总像素不超过 4000 万。
- 每件展品最多 6 张；API、数据库 attach / replace、归档导出 / link 验证与恢复共用 `media-policy.js` 的同一常量，更新已有图片不重复占位。
- 展示图最长边 1600 px，缩略图最长边 480 px。

浏览器先检查声明类型、文件字节数、数量和当前列表内的重复选择，但此时不解码不可信图片。原始二进制必须先通过服务端容器与像素上限校验，前端才继续创建 Canvas 派生图，避免在服务端拒绝像素炸弹之前主动触发浏览器完整解码。

### 2. 原图流式暂存与真实格式校验

`POST /api/media/uploads` 将二进制流写入独立 `.staging/upload-<uuid>`，同时计算 SHA-256 和限制字节数。服务端不信任扩展名或声明 MIME：

- PNG 校验签名、IHDR、IDAT/IEND 顺序、CRC、颜色/位深和尾随数据。
- JPEG 校验 SOI、段长度、尺寸帧、扫描结构和 EOI。
- WebP 校验 RIFF 长度、VP8/VP8L/VP8X 结构、尺寸和动画状态。
- SVG、GIF、伪 MIME、截断文件、非法结构和像素炸弹均被拒绝。

失败会删除整个上传暂存目录；应用启动时还会清理超过 24 小时的遗留 staging。

### 3. 派生图写入与完成

原图通过服务端校验后，浏览器才使用对象 URL 解码，并通过 Canvas 生成静态 WebP 展示图与缩略图；Canvas 派生不会携带原始 EXIF。异步处理带会话标识，切换展品、取消或销毁控制器时，旧结果不能写回新会话，对象 URL 和临时 Canvas 会及时释放。

前端分别通过 `PUT .../display` 和 `PUT .../thumb` 写入派生 WebP。服务端再次验证真实 WebP、静态状态、尺寸和字节限制；两个变体齐全前，完成接口返回 409。

`POST .../complete` 的顺序是：

1. 读取仍在暂存区的原图并尝试解析 EXIF。
2. 根据隐私模式确定内容哈希：保留原图时使用原图 SHA-256；仅安全图时使用展示图 SHA-256。
3. 若已有 ready 资产具有相同哈希，先逐项复验隐私策略、变体集合、磁盘字节数和 SHA-256；只有完整性通过才在同一独占操作中刷新 `updated_at` 宽限期、丢弃暂存并复用，损坏资产返回 409。
4. 否则将目录移动到 `assets/<hash-prefix>/<asset-id>/`，在同一外层 SQLite 事务内创建资产、变体和 EXIF observations。
5. `sanitized_only` 在移动前物理删除原图，只保留 `display.webp` 与 `thumb.webp`。
6. 数据库写入失败或并发产生重复资产时，最终目录先移入 `.trash` 再清理；若隔离重命名本身也失败，周期协调器会扫描并回收没有数据库记录的正式资产目录。

### 4. 关联、读取与编辑

图片完成并不自动进入某件展品。保存展品时，前端提交最终关联数组，由 `replaceMemoryMedia()` 在事务中统一更新封面、顺序、说明、替代文字、拍摄时间和照片背面。

图片通过 `GET/HEAD /api/media/:assetId/:kind` 读取：

- 返回前重新核对文件字节数与 SHA-256；内容被本地篡改时拒绝继续以旧 ETag 提供。
- 每个变体使用内容 SHA-256 ETag，支持 `If-None-Match → 304`。
- 展示图和缩略图使用私有 immutable 缓存；原图使用 `private, no-store`。
- 原图只有在用户选择保留原图且对应变体存在时才提供 URL。

卡片使用缩略图和数量摘要；详情胶卷使用展示图并保留说明、拍摄时间、照片背面和可选原图入口。图片加载失败有文字占位，不会让展品详情崩溃。

### 5. 解除关联与回收

直接删除仍被展品引用的资产会返回冲突。解除最后一个 `memory_media` 后，资产被标记为 `pending_delete`，文件目录隔离到 `.trash`，数据库删除成功后再物理清理。删除展品和启动 GC 只回收超过 24 小时宽限期的无引用 ready 资产，避免另一个表单刚完成上传却尚未关联时被并发删除。

清空馆藏先把所有媒体目录批量隔离，再提交 SQLite 清空事务；事务失败会反向恢复隔离目录，事务成功后才物理删除。物理删除失败会显式返回 `mediaCleanupPending`。完整 purge、归档导出 / 恢复关键区、上传完成、派生图写入、stale-stage 清理、隔离协调与 GC 共享同一进程内 FIFO，因而周期维护不能在慢 purge 的数据库提交前把 active quarantine 恢复回正式目录；维护调度本身采用 single-flight。协调器在启动时立即执行、本地运行时每 5 分钟重试：`.trash` 中仍有 ready / 被引用记录的目录恢复到原位置，已无记录或 `pending_delete` 的目录继续删除；正式 `assets` 中超过短宽限期且没有数据库记录的目录也先隔离再清除。这套“引用检查 → 隔离 → 数据删除 → 物理清理 → 启动 / 周期协调”流程覆盖隔离动作本身失败后的无主文件，避免把失败吞成成功或让原图墓碑失去追踪。

## EXIF：只产生待确认线索

`extractExifHints()` 只读取 JPEG APP1/Exif，并对 TIFF 字节序、IFD 条目数、偏移、类型、ASCII 长度和有理数执行边界检查；畸形输入返回空结果，不影响上传。

当前识别 `DateTimeOriginal`、`OffsetTimeOriginal`、GPS 与 Orientation：

- 所有 EXIF observation 初始状态都是 `suggested`，metadata 标记 `requiresUserConfirmation: true`。
- 有合法 offset 时保留显式时区；没有时区时保留 floating local datetime，不追加 `Z`。
- UI 只有在用户点击“采用这条线索”后，才把拍摄时间写入图片关联元数据，用户仍可继续修改。
- GPS 标记为敏感，不反向地理编码、不自动写 `memory.location`，普通 API 只暴露“存在敏感位置线索”而不返回坐标值。
- `sanitized_only` 不保存敏感 EXIF observation；`preserve_original` 下原文件可能继续包含完整元数据。

日期比较工具允许年/月/日不同精度：缺失保持 `unknown`，共同精度一致为 `compatible`，字面不同只进入 `review`，不会自动覆盖记忆日期。

## 相似图片：召回候选，不执行合并

前端从规范展示图绘制 9×8 RGBA 采样，服务端计算版本化 `dhash64-v1`。算法对透明像素固定合成白底，使用确定性亮度、重采样与 64 位比较规则，同时记录平均 RGB、亮度方差和宽高比。

服务端会校验采样尺寸、RGBA 字节数和编码格式，但由于本版刻意不引入服务端图片解码器，它不能证明客户端采样一定来自对应展示图。因此视觉指纹被标记为只用于召回的 system observation，不能升级为图片区域证据或事实来源；这也是候选永远需要人工核对、没有自动合并路径的另一层原因。

精确 SHA-256 相等是唯一的 `exact` 路径，并只用于存储资产复用。其余图片必须同时满足调用方固定策略：

- dHash 汉明距离不超过 8。
- 宽高比相对差不超过 0.12。
- 低亮度方差图片还需平均 RGB 距离不超过 32。

API 只返回 `similar_candidate`、`requiresReview: true` 和比较指标。公开指纹接口不返回原始 hash；实验台只有在用户点击“寻找可能相似照片”后才请求候选。系统没有确认、合并或删除候选的自动路径。

## OCR：本机选区、草稿和确认门禁

`media-ocr.js` 先检查浏览器本机 `TextDetector`：

1. 用户必须先拖拽圈选，或填写 `x/y/width/height` 规范坐标。
2. 模块只把这一范围裁入有尺寸和像素上限的临时 Canvas。
3. `TextDetector.detect()` 只接收本地 Canvas/ImageBitmap，不存在第三方 OCR 请求。
4. 结果写入最长 240 字的可编辑 textarea，并立即取消确认状态。
5. 用户核对文字与区域并勾选确认后，才 `POST` annotation API，保存为 `regionType: text` 的 `image_region`。

`TextDetector` 不存在、启动失败或未识别到文字时，界面明确说明限制并保留手动摘录，不会伪装成功。OCR 模块不调用记忆更新、AI 分析或讲解接口，因此没有自动改写展品正文的代码路径。识别与保存分别使用操作序号、会话号和 AbortController 隔离过期异步结果。

## 图片区域证据

`buildImageRegionObservation()` 对每条区域执行：

- memory/asset 关联校验和 ready 状态校验。
- `canonical-preview-v1` 下 0–1 坐标范围、边界和最小像素尺寸校验。
- 标签、备注、类型和敏感标记归一化。
- `sourceHash = sha256:<asset.contentSha256>` 来源锚定。
- `integrityStatus = source_verified` 与 `semanticStatus = user_confirmed` 分离。

区域增删改都经过专用 annotation API；Demo 中全部写入被统一拒绝。时光拼图读取已确认 observations，并返回两侧图片证据，来源哈希不匹配时可降级为 `source_invalidated`。

## 时光对照与手动两点变换

`media-compare.js` 默认只渲染可访问的两图并排视图。用户显式展开“手动叠影”后，左右图片各提供点 1、点 2，共两组对应点；鼠标点选和 0–1 数值输入具有同等能力。

当四点完整且两侧点对不重合时，前端由源向量与目标向量计算：

- 统一缩放 `targetLength / sourceLength`。
- 两向量夹角对应的旋转。
- 将右图第一个源点映射到左图第一个目标点的平移。

结果组成 Canvas 2D 相似变换矩阵，只包含平移、旋转和等比缩放。重合点、非有限值和过期图片加载不会产生伪参数；用户可调整透明度、撤销和重置。Canvas 失败时仍保留两图、说明、替代文字和坐标通道。模块不做特征检测、透视校正、自动地点识别或事件判断。

## 馆藏体检与只读备份验真

“数据与项目”里的馆藏体检默认折叠，启动后在媒体与声音互斥协调器内执行一项可取消的只读扫描：SQLite `quick_check`、外键、Schema/迁移账本、FTS 数量与成员集合分别核对；图片按隐私模式检查必需变体、内容锚点、文件字节数与 SHA-256，声音也重新核对实际文件。Schema 19 继续检查策展四表，并增加收件箱、来源护照、共忆记录及锁馆单例/校验和的 fail-closed 验证；结构损坏为 blocker，业务 `needsReview` 只标为 attention，不冒充数据库损坏。

扫描不会调用修复、删除或业务写入 API，同一时刻只允许一项；结果只存在当前进程内、最多公开 200 个最小化问题，并在 10 分钟后过期。停止操作通过 `AbortController` 结束后续核对，不把“已取消”伪装成通过。公开 Demo 禁用体检，避免共享实例读取或接收私人馆藏。

`POST /api/archive/inspect` 接收 `.time-isle` 流，复用正式恢复前的严格解包、manifest、哈希、格式、引用和 Schema 验证，只返回是否可恢复、格式/Schema 版本、条目/展开字节和各类安全计数，其中时间校准只返回条数，策展只返回运行/提案/决定计数。它不会暴露来源键、来源摘要、日期、目标、工具、提案或哈希，不物化到最终媒体目录、不打开业务事务，`finally` 总会删除本次 `.inspect` 暂存；future Schema 在任何业务写入前拒绝。

## `.time-isle` 严格归档

### 归档结构

完整归档使用确定性 POSIX ustar tar，再由 gzip 封装，主要条目为：

```text
manifest.json
collection.json
media/assets.json
media/links.json
media/media_observations.json
media/assets/<asset-id>/<kind>.<ext>
exhibitions/state.json
revisits/state.json
revisits/intents.json
entities/state.json
voices/state.json
voices/assets/<content-hash>.<ext>
oral-history/state.json
curator-agent/state.json
capsules/state.json
revisions/state.json
timeline/calibrations.json
```

`manifest.json` 记录格式版本、应用/Schema 版本、导出模式、功能 section 声明，以及每个业务条目的路径、MIME、字节数和 SHA-256。媒体、声音、修订、回访意愿、口述史、时间校准、受限策展审计、记忆收件箱、来源护照与共忆回信都以本次 collection 的闭包为边界。Schema 19 完整归档除既有 `timeline/calibrations.json`、`oral-history/state.json` 与 `curator-agent/state.json` 外，还必须显式包含 `inbox/state.json`、`provenance/state.json` 与 `co-memory/responses.json`；即使记录数为零也不能省略。口述史不复制声音字节，共忆 section 不复制目标展品正文。锁状态、salt、digest 和 verifier 不属于功能 section，完整/脱敏归档都不得携带。旧 Schema 不能越级声明新版数据，future schema 会在任何恢复写入前拒绝。

导出与 prepare / restore 共用发布上限：最多 500 件展品、500 项媒体资产、3000 条关联、10000 条图片线索、2000 个归档条目，单项 25 MiB、总展开 500 MiB；修订 section 另外限制为最多 500 件展品、10,000 条修订和精确 `20 MiB` UTF-8 JSON，恰好等于上限允许通过，超出一个字节即拒绝。调用者只能收紧限制，不能放宽后生成默认无法恢复的归档。

脱敏模式只保留无法反推个体内容的计数摘要：媒体索引/图片文件/observations、展览叙事与引用、实体名称/别名/关系、声音字节与文字稿、胶囊标题/日期/来源/快照/内部 ID 都物理排除；修订只保留展品数、修订数和固定说明；回访意愿只保留总数与固定说明。时间校准摘要严格只有固定五字段；口述史摘要严格只有 `mode / questionCount / answerCount / confirmedAnswerCount / note`；策展摘要只保留运行、完成/取消、提案、决定和批准/拒绝计数及固定说明。策展目标、来源 ID、工具名/参数/结果、提案、关系、动作、决定结果、内部 ID、哈希与时间戳物理缺席，并由专门隐私扫描器拒绝夹带。各 section 使用固定脱敏说明与精确键集，不能借 `note`、别名字段或额外数组夹带内容。分享编辑草稿从不持久化，完整和脱敏根对象都会拒绝 `shareDrafts` 等未知字段。

### 严格解包与全量验证

`time-isle-archive.js` 不调用系统 tar，也不接受宽松变体。解包会拒绝：

- 绝对路径、`..`、反斜杠、控制字符和暂存目录逃逸。
- 重复条目、NFC/大小写折叠后的路径碰撞。
- 符号链接、硬链接、目录和其他非普通文件类型。
- 非 ustar 头、错误校验和、非法数字、base-256、非零 padding、缺失结束块和尾随数据。
- 超过压缩字节、条目数、单项大小或总展开大小限制的归档炸弹。

`prepareMediaArchive()` 只在独立 staging 中工作，随后校验 manifest 精确覆盖、逐项哈希/MIME/大小、JSON 结构、图片/声音真实容器、资产/变体/关联/observation 交叉引用、展览/回访/实体/胶囊/修订/回访意愿/时间校准引用及条目上限。它还按 asset ID 建立隐私策略映射：EXIF source 只允许 `captured_at`、`orientation`、`gps_coordinates` 三种真实解析器输出，逐种校验状态、敏感标记、精确 value 键集和值域；`gps_coordinates` 必须是 `exif + sensitive + suggested`，`sanitized_only` 资产拒绝全部敏感 EXIF observation。胶囊 payload SHA、匿名快照、展示图引用与来源展览，修订规范快照、父链、类型和恢复引用，意愿精确键集、唯一展品引用、日期/时区组合与时间顺序，以及校准的 memory/event 互斥目标、分辨率、区间、最小来源快照、来源键、保存时摘要与导出时当前摘要，都在任何写入前验真。完整 `.time-isle` 恢复会从同一归档内的展品、修订、claim 与媒体 observation 重建候选，并要求重建摘要等于导出时当前摘要；归档来源闭包不一致时整包拒绝，而保存时摘要本来就不同的合法 `needsReview` 判断仍可原样恢复。归档导出、prepare、inspect、restore 规划和数据库写入均执行同一语义约束。prepare 只返回 `verified` descriptor，不写业务数据库或最终媒体/声音目录。

Schema 13 还会验证 `oral-history/state.json` 的精确键集、问题/回答数量、事件闭包、声音 ready 状态、片段不越过真实音频时长、状态时间顺序和脱敏计数。口述史引用的声音必须出现在同一归档的 `voices` section，字节不允许复制到口述史目录；任一问题、回答、事件或声音引用不完整都在物化前整包拒绝。

Schema 14 还会验证 `curator-agent/state.json` 的固定预算、运行/终态时间、步骤顺序、只读工具白名单、结果字节与哈希、来源闭包、提案重算、动作声明、逐项决定和幂等域。完整归档中的 memory/exhibition 引用必须落在同一 collection 边界；脱敏摘要必须通过独立隐私投影验证。任一回执、来源、提案或决定不一致都在物化前整包拒绝。

Schema 19 在上述合同之后继续验证 `inbox/state.json`、`provenance/state.json` 和 `co-memory/responses.json`。完整共忆记录必须落在同一 collection 展品边界，请求/回答摘要、memory anchor、来源键与 provenance snapshot 必须相互闭包；脱敏模式只接受各模块固定计数投影。任一问答、ID、哈希、anchor、来源集合或引用不一致都在物化前整包拒绝。归档注册表不包含 `museum_lock_state`，因此锁状态与 verifier 无法借完整模式越过排除边界。

### 事务恢复与文件补偿

恢复按依赖顺序执行：

1. 先规划 memory/asset 映射并恢复记忆、修订和媒体，再由记忆考古得到 event 映射，由声音恢复得到 voice asset 映射；相同内容 SHA-256 的 ready 媒体或声音资产可复用，冲突 ID 生成新 ID。
2. 使用 event + voice 两份映射恢复口述问题与回答，得到 question/answer 映射；随后恢复时间校准，使事件级 `oral-history` 来源键在 ID 重映射后仍与当前来源一致，再恢复展览并取得 exhibition 映射。
3. 恢复记忆收件箱来源、来源护照与共忆回信；共忆回信必须先取得旧到新 response/source/anchor 映射，引用它的 provenance source 才能建立。任何目标 memory ID 冲突或依赖缺失都会回滚整个事务。
4. 使用 memory/event/exhibition 映射恢复受限策展审计；所有 run/step/proposal/decision 生成新 ID，原请求和决定幂等键进入 `historical-*` 隔离域，运行强制为 `historical + needsReview + allowDecisions: false`。导入的批准只是历史回执，不调用保存、关系或发布副作用。
5. 已验证文件物化与上述数据库导入位于一个外层 SQLite 事务和文件补偿流程中；任何功能 section 条数不完整或出现 `skipped` 都会取消整次恢复。

如果任一步骤或数据库事务失败，SQLite 回滚，`cleanupMovedDirectories()` 反向删除本次已创建的媒体目录；请求 `finally` 还会删除恢复 staging。损坏归档在物化前就被拒绝，因此实现“损坏零写入”；数据库阶段失败则通过事务加文件补偿恢复到一致状态。按内容哈希命中本地 ready 资产时，恢复还会逐项比较隐私策略、变体集合、SHA-256、MIME、尺寸和字节数，并重新流式核对磁盘文件哈希；任一不一致都整批拒绝，不能用损坏或不同规范预览的本地资产替换健康归档副本。

进程异常退出可能来不及执行请求 `finally`。启动与周期维护因此只扫描 `.exports/time-isle-export-*`、`.restore/restore-*` 和 `.inspect/inspect-*` 三类契约目录，清除超过一小时的遗留暂存；新鲜任务、无关名称和不安全的符号链接父目录保持不动。它补的是崩溃后的明文暂存清扫，不替代逐请求清理。

完整 `.time-isle` 不包含早期文字整理的 `agent_runs/steps/events`，但显式包含受限策展审计与 schema 15–18 来源模块。旧 JSON 端点仍没有图片/声音二进制完整性语义；非空口述史继续返回 `requiresTimeIsle`，而收件箱、provenance 和共忆结构则必须先通过严格合同与引用验证，再在同一事务按依赖顺序恢复。普通 JSON 与 `.time-isle` 都明确排除锁状态和 verifier，避免在新设备恢复馆藏时意外复制写保护口令材料。两种格式仍在任何写入前拒绝 future schema；需要无损迁移媒体、声音、口述史与完整来源闭包时必须使用 `.time-isle`。

## 记忆整理、检索与考古

`POST /api/analyze` 先限制原文长度；非 Demo 模式配置 `AI_API_KEY` 时调用 OpenAI-compatible Chat Completions，未配置、请求失败或处于 Demo 模式时使用本地 Mock。模型 JSON 会被归一化为允许字段，同一结果保存为档案、策展、编辑三阶段快照；它不是三个自治 Agent。用户编辑并确认后才保存展品。

混合检索对标题、展品说明、原文、人物、地点、标签、情绪和来源加权，返回命中字段与词。讲解员只接收 Top-K 馆藏摘要，并把同批展品作为 citations 返回；当前不将其包装成生产级向量 RAG 或程序级忠实度保证。

记忆考古继续遵循既有证据边界：

1. 航线按人物、地点、日期、标签、情绪和有限关键词加权，所有关系初始为 `sameEvent: unassessed`。
2. 文字拼图证据带 `sourceQuote/start/end`，可由 `rawContent.slice(start, end)` 复核；位置正确不代表记忆为真。
3. 一侧缺失进入补充或未知，不当作矛盾。
4. 用户可以回答单个补问、跳过或保留不确定，回答不自动修改原文或字段。
5. 只有人工确认 API 才创建事件版本组；事件、pair decision 和 claims 同事务写入。
6. 编辑原文时重新核验历史 claims，找不到旧摘录则标记 `source_invalidated`。

影像阶段在拼图响应中增加左右两侧 `imageEvidence` 和媒体集合，再由时光对照提供目视工具；V5–V7 的展览、线索、声音与胶囊也没有改变“用户最终确认”的事件边界。

## 安全与隐私

- `.env`、`data/`、日志、临时 staging 和调试导出不进入 Git。
- CSP 仅为本地图片交互增加 `blob:`，不增加远程图片域；响应同时包含 `nosniff`、frame 限制和 Permissions Policy。
- 图片不会发送给 AI 整理或讲解接口；本机 TextDetector 没有远程后备。
- 原图、展示图、缩略图与声音响应都使用 `private, no-store`，不会进入 PWA 私人馆藏缓存。
- 完整归档包含原文和媒体，属于敏感文件；脱敏归档物理排除媒体。
- 胶囊日期是仪式门槛；未到期接口不读取 payload，但本机数据库本身仍依赖设备磁盘保护。
- 离线文件口令不上传、不持久化；应用无法找回口令，也不在文件中保存验证提示或明文摘要。
- 分享候选默认不代表同意；V2 只包含逐项选择的引用、display WebP 和 confirmed 文字稿，并物理排除内部 ID/URL/SHA、未选原文与原始声音二进制。
- 回访意愿只来源于明确确认，不保存原因或推断心理；它不影响搜索、展品读取和用户主动浏览。
- 时间校准只读取四类受控来源，GPS 被严格排除；判断与来源分表保存，`needsReview` 仅表示来源集合变化，不等于事实错误，也不会触发日期回写。
- 受限策展助手不读取环境中的模型 Key，不访问网络/文件/任意 SQL；只读工具回执仍可能包含私人证据，完整审计属于敏感数据，分享投影与脱敏归档物理排除它。
- 共忆邀请与回信在馆外由浏览器端加密，但确认入馆后的问答、身份标签与来源锚点是 SQLite 明文；加密不证明身份，文件也没有数字签名。
- 设备内镜片只按 ID 重读保存态并运行确定性规则，固定零外部模型、零工具和零保存；客户端不向预览 API 上传记忆正文。
- 锁馆只阻止新的应用写请求，verifier 不等于明文口令且不进入普通归档；SQLite、FTS、图片、声音和磁盘仍未静态加密。
- 结构演练只在暂存区验 manifest、哈希与引用，不执行当前馆藏恢复或隔离恢复，不能作为灾难恢复证明。
- 清空本地馆藏要求正文 `confirm: DELETE`。
- Vercel `/tmp` 只适合 Demo，不作为持久化承诺。

## Demo 模式

`INTERVIEW_DEMO=true` 时：

- SQLite 与 `MEDIA_ROOT` 切换到系统临时目录，并在冷启动注入四件示例展品、1 场可用于三层分享流程的已确认展览，以及由两件校园记录组成的确认事件和一项 `alternatives` 时间校准。
- Demo 模式在代码层强制选择 Mock；即使进程环境误配 `AI_API_KEY`，整理和讲解也不会调用外部模型。
- 共享文本馆藏、整理运行、时光拼图与补问分别设置固定总量上限；同步复检与写入位于同一 SQLite 事务，并发达到上限后返回 429。
- 所有媒体 mutation 在 `media-api.js` 的统一入口返回 403，包括上传、完成、指纹、关联、删除和 annotation 写入。
- 声音、文字稿、实体别名/合并、展览持久化、回访状态、长期回访意愿、时间校准与胶囊写入均返回 403；只保留不会落库的预览、只读校准来源和本地浏览器三层加密演示。
- 口述史 GET 只读返回符合条件的问题与已有来源；浏览器不请求麦克风、不打开文件选择、不上传声音，也不发送口述史 PUT/DELETE。没有真实示例回答时不伪造可播放音频。
- 策展助手 GET sample 由 API 内存合成，不读取或写入 store；全部非 GET/HEAD 请求在请求体、ETag、幂等键和持久化之前返回 403，运行列表保持空，不播种真实审计记录。
- 线上 V14 受保护 Demo 中，共忆文件输入/确认、锁馆转换和结构演练上传都在请求体读取前返回 403；镜片只允许对合成/播种展品做确定性只读预览，不接收私人正文副本。
- 服务端另外阻止全部 DELETE、清空、导入、完整归档恢复、备份验真、馆藏体检和展品后续 PUT，避免共享访客制造修订或提交私人归档。
- 讲解、回顾、记忆航线等只读路径继续可用；整理与讲解始终使用 Mock。
- Demo 状态和隐私接口明确提示共享临时实例、禁止私人内容。

Demo 不是账号隔离环境，临时纯文字新增可能被同一实例看到并随实例重建丢失。上述 V11–V14 新模块与高风险操作的只读 / 禁写边界已部署到公开网址；2026-07-20 的生产复核确认 `/api/version` 与 `/api/health` 返回 `14.0.0 / schema 19 / interview-demo / ephemeral-sqlite / mock-fallback`，策展 sample 为 `synthetic + demo` 且 runs 为 0。锁馆与结构演练的虚构 `text/plain` 探针均在读取正文前返回 `403 / MUSEUM_LOCK_DEMO_READ_ONLY / bodyBytesRead: 0`，前后 stats、锁状态和 runs 不变。生产事实只证明本次实际请求的 sample、锁馆和结构演练三项边界；其余高风险保护来自发布代码合同与本地整库回归。共忆确认、真实锁馆转换、结构验真和其他完整可写能力仍只应在隔离本地 V14 中使用虚构数据演示。

## 诚实限制与后续边界

- `TextDetector` 并非普遍可用；当前没有第三方 OCR 或随项目分发的中文语言包。
- EXIF 不自动反查地点，相似候选不自动合并，区域标签和时光对照不做事实判断。
- 手动两点只支持二维相似变换，不是自动配准或计算机视觉定位。
- 当前无账号、云数据库、云对象存储和跨设备同步。
- 记忆年轮的 SHA-256 父链用于发现快照或顺序被破坏，不含外部签名、可信时间戳或远端见证；能控制本机数据库的人仍可整体重写账本。
- 本地 SQLite、图片和声音目录没有内建静态加密；日期未到只能阻止正常 API 读取，不能替代设备磁盘加密和系统账号保护。
- 加密离线展览依赖现代浏览器 WebCrypto；口令无法找回，下载后无法远程撤回，知道口令的人仍可复制、转发或截图；原始音频不进入分享文件。
- 回访意愿是显式产品偏好，不是心理画像；welcome 只做合格候选内优先，later/pause 只约束主动回访，不影响搜索或用户主动查看。
- 时间校准是用户确认的展示判断，不是事实证明或“真相评分”；来源键和 `sourceSetSha256` 用于发现来源变化，不证明来源内容真实。JSON 缺少媒体/EXIF 时会提示使用 `.time-isle`，不会猜测性重建来源。
- 口述史文字稿、说话人和时间含义不由系统推断；当前没有自动转写、说话人识别或情绪分析。非空口述史 JSON 不能代替包含声音字节的 `.time-isle`。
- 当前是可安装的响应式 PWA，不是原生移动 App；手机与桌面浏览器共用同一套四项导航，安装只增加启动入口，完整馆藏仍依赖对应数据服务，不承诺私人数据离线同步。
- 真实模型讲解没有逐句引用解析和固定标注集忠实度评估。
- V10 受限策展助手已经实现，但它只是确定性的本地证据执行器：重放评测验证预算、回执与提案一致性，不衡量审美质量，也不证明关系或展览事实正确；保存、关系确认、发布与分享仍须逐项人工决定。
- 共忆信笺的 AES-GCM 只保护馆外文件内容，回信身份仍未验证且文件未签名；确认入馆后的问答是 SQLite 明文，也没有远程撤回、身份认证或口令找回。
- 设备内镜片不是向量检索、embedding 或本地大模型；它只按固定字段规则生成一次性预览，不能解释未保存的信息或证明关系与事实。
- 锁馆是应用写门禁，不是 SQLite、媒体、备份或磁盘静态加密；拥有设备和文件访问权的人仍可能绕过应用读取或修改数据。
- 结构恢复演练没有执行当前馆藏恢复或隔离恢复，只验证当前归档合同；它不能替代在另一设备上实际恢复、离线备份轮换和灾难恢复演练。
- 若未来增加云同步、向量检索或模型视觉能力，必须继续遵循显式授权、可撤回、来源可追踪和人工确认原则，且不能重新挤占默认首页。
