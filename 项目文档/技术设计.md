# 技术设计

## 架构概览

```text
Browser
  ├─ public/index.html
  ├─ public/styles.css
  ├─ public/archaeology.css
  └─ public/assets/app.js
            │ fetch JSON
            ▼
server.js / api/index.js
  ├─ 记忆 CRUD
  ├─ AI 整理与 Mock fallback
  ├─ 混合检索、讲解与回顾
  ├─ 记忆航线、时光拼图与人工确认
  ├─ 导入导出与隐私边界
  └─ Demo 安全限制
            │
            ▼
lib/archaeology.js（只读证据算法）
database.js → SQLite（原文、版本组、决定与补问）
```

本地运行由 `server.js` 启动 Node HTTP Server；Vercel 环境由 `api/index.js` 导出同一个 request handler，避免维护两套路由。

## 数据模型

主要表：

- `memories`：原文、展品说明、展厅、日期、地点、重要度和媒体备注。
- `memory_people`、`memory_tags`、`memory_emotions`：多值字段，使用外键关联展品。
- `agent_runs`：一次整理的模式、状态、摘要和关联展品。
- `agent_steps`：档案、策展、编辑等步骤的输出与证据。
- `agent_events`：工作流开始、结构化、等待确认等事件。
- `memory_events`、`event_members`：用户确认后的同一往事及其多个独立版本。
- `memory_claims`：从拼图中持久化的字段、原文引用与字符位置；数据库会再次校验引用是否合法。
- `memory_pair_decisions`：按规范化 pair key 保存用户对两个版本的决定。
- `curator_questions`：补充问题、回答、跳过或保留不确定的记录。

升级已有 SQLite 时，旧版专题展和报告草稿表可能仍留在文件中；自 V2 起不再读取或写入这些表，V3 继续保留该兼容策略。为避免擅自销毁历史数据，升级过程不主动删除旧表。

## 记忆整理

`POST /api/analyze` 的处理顺序：

1. 限制原始文本长度并校验输入。
2. 有 `AI_API_KEY` 时请求 OpenAI-compatible Chat Completions。
3. 验证模型 JSON，并把字段归一化到允许的展厅、范围和长度。
4. 未配置 Key 或请求失败时，使用本地关键词规则生成草稿。
5. 将同一份模型/规则结果组织成档案提取、策展标注、草稿生成三阶段处理快照，并先保存 run、step 与 event。它不是三个独立模型 Agent 的自治协作。
6. 前端允许用户修改；确认后 `POST /api/memories` 保存展品并关联 run。

模型失败不会阻断核心流程，也不会把错误响应直接写入馆藏。

## 检索与讲解

SQLite 层读取馆藏后，对下列字段进行加权匹配：

- 标题、展品说明、原始记忆。
- 人物、地点、标签、情绪和来源。
- 少量中文语义词组扩展。

搜索结果返回分数、命中字段、命中词和置信提示。讲解员只接收排名靠前的展品摘要，并把相同展品作为 citations 返回前端。

这是一种适合小型个人馆藏的可解释检索方案，不把它包装成生产级向量 RAG。

## 记忆考古

记忆考古完全复用已经确认保存的馆藏，不会在记录阶段自动合并内容：

1. `buildConnections()` 对人物、地点、日期、标签、情绪和有限原文关键词加权，最多返回少量关联以及逐条理由。
2. 每个关联固定标记为 `sameEvent: unassessed`，表示尚未判断是否属于同一往事。
3. `buildPuzzle()` 比较两个候选版本。日期、地点、人物、标签和情绪等字段级证据必须带 `sourceQuote/start/end`，并能由 UTF-16 字符区间 `rawContent.slice(start, end)` 复核；这只验证摘录位置，不验证记忆真假。
4. 一侧缺失只进入“补充”或“未知”，不会被归为矛盾。
5. `buildCuratorQuestion()` 按“差异优先于补充、补充优先于未知，再按日期/地点/人物等固定权重”只选择一个问题；用户可以回答、跳过或明确保留不确定。回答作为独立记录保存，不自动修改原文、字段或 Claims。
6. 只有 `POST /api/archaeology/events` 才会建立事件版本组。事件、pair decision 和两侧 claims 在同一个 SQLite 事务中写入，失败时整体回滚；本地模式也允许用户解除版本分组。
7. 展品原文被编辑时，数据库会在同一保存事务中重新核验历史 Claims；旧摘录找不到时改为 `source_invalidated`。

算法层位于 `lib/archaeology.js`，不调用模型且零外部依赖，因此本地模式和公开 Demo 都有确定性结果。未来若增加模型抽取，模型输出仍必须通过同一原文位置校验。

## 数据安全

- `.env`、`data/`、日志和调试导出均被 Git 忽略。
- 馆藏与记忆考古备份包含原文、事件版本组、Claims、pair decisions 和补问；不包含 `agent_runs/steps/events`。脱敏导出会隐藏原文、人物、地点、附件、版本关系和补充回答。
- 清空本地数据库需要正文 `DELETE` 确认。
- Demo 模式会阻止所有 `DELETE`、导入和预置展品修改；新增示例内容、临时拼图确认和补问仍可能被同一实例的其他访客看到。
- 响应包含 CSP、`nosniff`、禁止 iframe 和权限限制等基础安全头。

## Demo 模式

启用 `INTERVIEW_DEMO=true` 后：

- 数据库位置切换到系统临时目录，或使用显式 `DB_PATH`。
- 冷启动时注入四件示例展品和一条 Agent run。
- 删除、导入与预置展品修改返回 403；新增示例和临时拼图仍可体验。
- 未配置真实 Key 时使用 Mock fallback。

Vercel `/tmp` 是临时且可能多实例的，因此 Demo 只用于功能体验，不作为真实用户数据存储。

## 测试策略

`scripts/frontend-check.js` 检查：

- 四个主视图和核心操作是否存在。
- 航线只作为局部回顾标签，拼图通过独立 dialog 渐进披露。
- 页面是否重新出现 Phase/Reviewer/插件治理术语。
- 是否存在内联脚本、渐变或失控的核心文件规模。
- 前端和服务端是否仍覆盖核心 API。

`scripts/api-smoke.js` 使用临时 SQLite 启动真实服务，覆盖：

- 静态资源与安全头。
- Mock 整理、Agent run 保存与展品关联。
- 展品 CRUD、检索、讲解、回顾、导入导出。
- 航线理由、拼图原文锚点、人工确认版本组和“保留不确定”。
- 清空确认。
- Demo 示例注入、临时存储和删除保护。

`scripts/archaeology-check.js` 另外覆盖强弱关联、原文区间复核、缺失不当矛盾、两位数日期、单问题生成和空馆藏；完整 HTTP smoke 当前共 53 条断言，并覆盖版本组确认/解除、编辑后证据失效、补问保留、馆藏与考古备份恢复、损坏备份拒绝及 Demo 预置数据保护。测试强制清空 `AI_API_KEY`，因此不覆盖真实模型回答质量。

## 后续扩展方向

只有在核心路径稳定且出现真实需求后，再考虑：

- 用户认证与托管数据库。
- 向量检索或 embedding 缓存。
- 图片和语音的受控上传。
- 可选择的端到端加密同步。
- 在固定人工标注集上评估事件候选的 Precision、Recall、F1 与误合并率。

这些能力不应重新进入默认首页，也不应以阶段编号替代用户价值。
