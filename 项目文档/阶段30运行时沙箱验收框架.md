## 3. Fixture 场景

| Fixture | 目标 | 预期结果 |
| --- | --- | --- |
| allow-basic-manifest-fixture | 验证最小 manifest 可被静态读取 | pass-dry-run |
| deny-network-access-fixture | 验证网络访问请求被拒绝 | blocked |
| deny-secret-read-fixture | 验证插件密钥读取被拒绝 | blocked |
| deny-sqlite-memory-read-fixture | 验证 SQLite 私人记忆数据不可达 | blocked |
| timeout-runaway-fixture | 验证超时场景进入 quarantine | quarantined |
| invalid-output-fixture | 验证不合规输出进入 quarantine | quarantined |
| crash-exit-fixture | 验证异常退出不会写入运行时状态 | quarantined |

所有 fixture 都是描述性输入，不包含真实第三方插件代码。

## 4. 隔离边界

验收框架必须确认以下边界：

- no-network-access
- no-secret-access
- no-sqlite-memory-access
- no-host-filesystem-write
- no-runtime-state-write
- no-release-state-write
- no-third-party-code-execution
- quarantine-only-output

任一边界被破坏时，结果必须是 `blocked` 或 `quarantined`，不能是 `pass`。

## 5. 证据格式

每个 fixture 的 dry-run evidence 应包含：

- `fixtureId`
- `intent`
- `expectedResult`
- `observedResult`
- `boundaryChecks`
- `quarantineRequired`
- `runtimeExecution`
- `thirdPartyExecution`
- `releaseReady`
- `phase30EntryReady`
- `manualReviewRequired`

当前阶段所有 evidence 都必须保持：

- `runtimeExecution=false`
- `thirdPartyExecution=false`
- `releaseReady=false`
- `phase30EntryReady=false`

## 6. 接受标准

沙箱验收框架通过的含义仅限于“验收设计完整”。它不表示可以执行第三方插件。

接受标准：

- fixture 场景覆盖允许、拒绝、超时、输出隔离和异常退出。
- 所有敏感边界都有明确拒绝规则。
- quarantine-only-output 被固定为默认结果路径。
- pass-dry-run 不能推出 runtime 或 release 状态变化。
- blocked 和 quarantined 结果不能被人工签核自动覆盖。
- 检查脚本只验证文档和设计完整性，不执行任何插件代码。
