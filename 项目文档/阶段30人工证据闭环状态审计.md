## 2. 新增命令

```bash
npm.cmd run phase30:evidence-closure-status
```

该命令读取未来的 `data/phase30-human-evidence-submission.json`。如果文件不存在，命令会输出 `no-live-submission-status`，并列出下一步应继续收集真实 reviewer 输出。

如果文件存在，命令会检查：

- 10 个必需 evidence slots 是否都存在。
- `reviewer`、`reviewedAt`、`evidenceRef`、`disposition`、`decisionReason`、`residualRisk` 是否已由真人填写。
- 是否仍含 `REPLACE_WITH_...`、`YYYY-MM-DD`、`pending`、`missing`、`unassigned` 等占位符。
- `accepted-with-risk`、`deferred-with-owner` 或 `blocked` 是否补齐 `followUpOwner`、`recoveryCondition`、`targetReviewDate`。
- 是否存在重复 evidenceId。
- 是否仍保持 `releaseReady=false`、`phase29ExitReady=false`、`phase30EntryReady=false`、`runtimeExecution=false`、`thirdPartyExecution=false`。

该命令已进入 `npm.cmd run check`，用于保证人工证据闭环状态审计不会漂移。

## 4. 推进顺序

1. 使用 `阶段30人工Reviewer交接包.md` 把 10 个 slots 分给真实 reviewer。
2. 由 reviewer 在线下提供真实 `reviewer`、`reviewedAt`、`evidenceRef`、`disposition`、`decisionReason` 和 `residualRisk`。
3. 用 `阶段30会后证据汇总包.md` 汇总真实 reviewer 输出。
4. 运行 `phase30:evidence-closure-status`，确认缺口清单。
5. 在所有 slots 完整且维护者明确允许转换前，继续保持 live submission 不存在。
6. 满足转换条件后，再运行 `phase30:live-submission-conversion-preflight`。
7. 转换为 live submission 后，运行 `phase30:evidence-submission-gate`。
