## 1. 项目简介

AI 记忆博物馆是一个娱乐型 AI Agent 项目。它把用户输入的日记、聊天片段、照片描述、旅行片段、梦境或普通生活瞬间，整理成一座有温度的私人虚拟博物馆。

用户不是简单地和 AI 聊天，而是在和一组 Agent 一起整理自己的记忆资产：档案员负责提取线索，策展人负责归类，编辑负责生成展品文字，讲解员负责基于已保存记忆进行导览和问答。

当前版本已经从早期纯前端 Mock 原型，推进到 Node 后端、SQLite 持久化、真实 AI 接口、Agent 运行历史、混合 RAG、多模态附件线索、时间线、主题展、回忆报告、轻量工作流编排、数据主权控制、产品化运维信息、工程模块化、第十五阶段可保存专题资产、第十六阶段本地优先同步闭环、第十七阶段真实多设备同步适配层、第十八阶段长期记忆助理、第十九阶段外部资料导入和第二十阶段平台插件边界并存的状态。第二十阶段第十一版先固定插件生态的扩展点、内置插件注册表、权限复核、审计日志、安全策略、扩展契约测试、沙箱边界、无代码模板包、模板预览 fixtures、签名 manifest、插件安装流程模型和 readiness 检查，暂不执行第三方插件代码。

## 2. 项目愿景

每个人都有很多零散的记忆：

- 一次毕业后的沉默
- 一段朋友之间的聊天
- 一张老照片背后的故事
- 一次失败后重新开始的晚上
- 一件很小但很难忘的事情

AI 记忆博物馆希望把这些零散内容保存、整理、命名和展示出来，让用户像参观博物馆一样重新认识自己的过去。

项目追求的体验不是效率工具式的冷冰冰归档，而是“可保存、可回看、可讲述”的温柔整理体验。页面布局、展品卡片、馆藏编号、复核状态和讲解员导览都围绕“私人展馆”的隐喻展开。

## 3. 核心用户

适合用户：

- 喜欢记录生活、整理日记、回看相册的人
- 想把个人记忆转化成展览式内容的人
- 想体验 AI 创意应用的人
- 想做 AI Agent 项目的新手开发者
- 想学习前后端、SQLite、AI API、RAG、Prompt 和 Agent 工作流的人

## 5. Agent 设计

当前系统包含四个主要 Agent 角色。

### 档案员 Agent

负责从原始记忆中提取结构化信息，例如人物、地点、时间、来源和原始线索。它决定一段记忆进入系统时最基础的档案字段是否完整。

### 策展人 Agent

负责判断记忆应该进入哪个展厅，并生成情绪、情绪强度和珍藏级别等策展信息。它让一段私人记忆从普通文本变成有分类、有语境的展品。

### 编辑 Agent

负责生成展品标题、标签和展品说明。它把原始记忆压缩成适合展示和回看的导览文字，同时保留私人记忆的情绪质感。

### 讲解员 Agent

负责和用户对话，带用户参观展厅，并基于已保存展品进行 RAG 检索问答。当前讲解员已经接入 `/api/guide`，可以从数据库中召回候选展品再回答。

## 6. 技术架构

当前项目采用轻量级本地优先架构：

- 前端：HTML、CSS、原生 JavaScript
- 后端：Node.js HTTP 服务
- 数据库：SQLite，使用 Node.js 内置 `node:sqlite`
- AI 接口：OpenAI-compatible Chat Completions
- 配置方式：`.env` 和环境变量
- 数据回退：后端 Mock、前端 Mock、localStorage 本地备份
- RAG：基于 SQLite 字段权重、中文关键词抽取和本地语义线索扩展的轻量混合检索
- Agent：显式 workflow steps + summary，而不是黑盒单次生成
- 检查：`npm.cmd run check` 覆盖语法检查、第十一阶段 readiness 检查和临时后端 API smoke test
- Smoke test：`npm.cmd run smoke` 会启动临时后端和临时 SQLite，验证 `/api/health`、`/api/analyze`、`/api/search?mode=keyword|semantic|hybrid`、`/api/guide`、`/api/agent-runs/:id` 与展品整理历史关联接口

这个技术路线优先保证项目可运行、可理解、可迭代。后续如果产品复杂度继续提高，可以再引入 Express、React/Vite、向量数据库或 LangGraph.js。

## 9. 后续方向

项目接下来不只继续添加页面功能，而是逐步把“记忆博物馆”做成一个可长期使用、可追溯、可扩展、可部署的私人记忆 Agent 产品。后续路线以本地优先、温馨体验、可解释 AI 和数据可控为核心原则。

### 阶段 7：后端可持久化 Agent 运行记录

把第六阶段的前端状态机升级为后端可记录的 Agent 运行历史。每次整理记忆都应产生 run id、步骤状态、输入输出摘要、失败原因、重试记录和人工确认记录。用户可以回看一件展品当初是怎样被 Agent 整理出来的。

### 阶段 8：语义检索和混合 RAG

把当前关键词检索升级为“关键词 + 向量语义”的混合检索。讲解员不只根据字面词召回展品，还能理解“那次我很失落但被朋友拉了一把”这类模糊问题，并展示引用来源、匹配理由和可信度。

### 阶段 9：多模态记忆展品

支持照片、截图、语音转写和更丰富的附件。照片可以被整理成展品封面，OCR 可以提取截图文字，图片描述可以成为记忆线索。博物馆从文字展品扩展为真正的生活档案空间。

### 阶段 10：时间线、主题展和回忆报告

在单件展品之外，系统开始理解一组记忆之间的关系。用户可以按月份、年份、人物、地点或情绪生成主题展，也可以得到年度回忆报告、人生阶段回顾和“那些反复出现的关键词”。

### 阶段 11：更正式的 Agent 编排引擎

当 Agent 流程开始包含分支、暂停、恢复、驳回、重试和跨展品分析时，引入更明确的工作流引擎或评估 LangGraph.js。目标不是追求复杂框架，而是让多 Agent 协作更稳定、更容易测试。

### 阶段 12：账号、多端同步、隐私和数据主权

如果项目从本地原型走向真实使用，需要补上账号体系、多端同步、导出删除、备份恢复、敏感字段保护和可选加密。私人记忆产品必须优先保证用户知道数据在哪里、如何被使用、如何彻底带走。

### 阶段 13：产品化、部署和运维

最后把项目从可运行原型推进到可交付产品：前端工程化、后端服务化、部署脚本、日志观测、错误追踪、测试矩阵、演示数据、使用文档和版本发布流程。此阶段重点是稳定性、可维护性和可演示性。

### 阶段 14：工程模块化和服务边界重构

第十三阶段已经证明项目具备本地交付能力，下一步应拆分后端 routes、services、agents、rag、workflow、privacy、operations、exports，以及前端 state、api、renderers、panels、forms 等边界。此阶段重点不是换框架，而是先把职责拆清楚，让后续深度功能不被单文件规模限制。

### 阶段 15：专题资产、报告编辑和可保存展览

把时间线、主题展和回忆报告升级为可保存资产。用户可以把一组展品整理成“家庭饭桌主题展”“毕业季主题展”“年度回忆报告”，并继续编辑章节、引用、封面和导览词。

### 阶段 16：真实多端同步和冲突处理

把当前手动 JSON 备份推进到更真实的多端同步模型。重点是设备 ID、变更日志、同步批次、冲突预览和冲突解决，而不是直接上云覆盖用户数据。

### 阶段 17：真实多设备同步适配层

在第十六阶段“手动同步包、冲突处理、风险确认、恢复演练和同步审计”的本地优先底座上，先建立真实多设备同步的适配层。此阶段重点是设备注册表、同步队列、通道抽象、设备信任策略和安全边界；手动 JSON 通道保持可用，局域网桥接和私有云适配先作为 planned 能力，不自动发现设备，也不自动上传私人数据。

### 阶段 18：Agent 能力进阶和长期记忆助理

让 Agent 从“整理单条记忆”升级为“长期协助管理记忆资产”。它可以发现缺失字段、推荐主题展、生成周期回顾、识别跨展品关系，并根据用户采纳或驳回持续改进。

### 阶段 19：个人知识生态和外部导入

支持从 Markdown、CSV、JSON、聊天记录、日记、备忘录、照片目录等来源批量导入，形成个人生活资料的整理入口。导入过程需要预览、清洗、拆分和来源追踪。

### 阶段 20：可扩展产品平台和插件生态

当核心体验稳定后，把记忆博物馆推进为可扩展平台。插件可以接入导入器、导出器、Agent 工具和专题展模板；部署形态可以扩展到本地桌面、局域网和私有云。

近期优先级继续推进阶段 20：第十一版已经完成平台边界、manifest schema、权限复核、审计日志、内置插件注册表、扩展契约测试、沙箱边界、无代码模板包、模板预览 fixtures、签名 manifest、插件安装流程模型、前端面板、导出结构和检查覆盖。下一轮应优先定义插件复核工作台、插件 lockfile 和发布前签名核验细则；真实第三方插件运行时需要等沙箱、密钥管理和审计闭环成熟后再启用。

后续每次项目方向、功能范围、技术路线、阶段状态或核心体验发生变化，都要更新本白皮书。
