# 仓库索引迁移执行方案

更新日期：2026-07-05

Plan: `git-flattening-migration-plan`

## 1. 执行边界

当前已完成路径扁平化迁移提交，但不是 release 阶段：

- `gitPathShape=accepted-flattened-project-root`
- `gitIndexMigration=committed-flattened-project-root`
- `docsOwnershipDecision=docs-in-repo`
- `indexMutation=committed-path-migration`
- `releaseCommit=blocked-until-human-release-approval`

本方案不批准 release，不退出 Phase 29，不进入 Phase 30 implementation，不启用 runtime execution，也不启用 third-party execution。

## 2. 迁移对象分类

### 2.1 工程代码路径

旧索引路径 `项目工程/...` 已有当前仓库根目录对应文件，例如：

- `项目工程/server.js` -> `server.js`
- `项目工程/app.js` -> `app.js`
- `项目工程/scripts/...` -> `scripts/...`
- `项目工程/src/...` -> `src/...`

这部分在获得明确授权后，可以作为“路径扁平化”迁移对象处理。

### 2.2 文档路径

旧索引路径里仍有 `项目文档/...` 删除项，但当前文档目录位于仓库根的上一级：`../项目文档`。

当前文档目录状态：`outside-current-git-root`。

这意味着它不能被当前 git root 直接纳入提交。后续必须先做文档归属决策，不能在未说明的情况下把旧文档删除混入迁移提交。

可选决策：

- `docs-external`：接受 `项目文档` 作为仓库外工作文档，旧索引文档路径在迁移提交中移除。
- `docs-in-repo`：先把文档目录复制或迁移到当前 git root 内，再纳入版本控制。
- `repo-root-reopen`：重新讨论仓库根是否应回到上一层；这与已接受的扁平化方向冲突，当前不推荐。

当前决策：`docsOwnershipDecision=docs-in-repo`，文档进入当前 git root 内的 `项目文档/`。

## 3. 授权后执行顺序

只有维护者明确授权“执行索引迁移”后，才允许进入以下动作：

1. 运行 `node scripts/git-flattening-preflight.js`。
2. 运行 `node scripts/git-flattening-migration-plan.js`。
3. 先解决 `docsOwnershipDecision`。
4. 只处理路径迁移，不混入 release、Phase 30 entry、runtime enablement 或第三方插件执行。
5. 迁移后运行 `npm.cmd run check`。
6. 审阅 `git status --short` 和 `git diff --cached --stat`。
7. 再由维护者决定是否创建单独的路径迁移 commit。

## 4. 当前禁止事项

- 不执行 `git add -A`
- 不执行 `git add -u`
- 不执行 `git rm`
- 不执行 `git commit`
- 不执行 `git reset --hard`
- 不执行 `git checkout --`
- 不执行 `git clean`
- 不把 check 通过解释成人工签核
- 不把 dry-run 通过解释成 runtime 准入

## 5. 文档归属决策更新

已接受 `docs-in-repo`。后续迁移以当前 git root 内的 `项目文档/` 作为文档版本控制位置。

当前仍不是索引迁移执行：

- `docsOwnershipDecision=docs-in-repo`
- `outsideMirrorRetained=true`
- `indexMutation=committed-path-migration`

仓库外 `../项目文档` 暂时作为镜像来源保留，不在本轮删除。

## 6. 当前结论

当前可以继续做提交后审查，但不应执行 release commit。最关键的剩余人工动作是继续补齐 Phase 30 人工证据和 release 审批材料。

不执行 release commit。
