# 时屿（TIME ISLE）｜简历项目描述

## 推荐版

**时屿（TIME ISLE）｜AI 辅助的私人记忆策展与证据回溯工具｜Node.js · SQLite · Vanilla JavaScript · Vercel**

- 独立开发本地优先的记忆整理应用，将日记、聊天片段和照片文字描述转换为可编辑的结构化展品，支持人物、地点、日期、标签、情绪与重要度管理。
- 将一次模型调用（无 Key 时为本地规则解析）组织为档案提取、策展标注、草稿生成三阶段可追踪工作流，持久化 run、step 与 event；分析轨迹先行记录，草稿仅在用户确认后写入馆藏。
- 实现字段加权检索与引用式讲解，将 Top-K 展品作为回答上下文，并返回命中字段、规则强度和同批检索来源；语义扩展采用本地词表，不依赖向量数据库。
- 实现零依赖的确定性记忆航线与双片段对照算法，对人物、地点、日期、标签和情绪进行加权关联；字段级证据携带原文摘录和 UTF-16 字符区间，缺失信息不判为描述差异。
- 由用户确认或解除事件版本组，在同一 SQLite 事务中写入成员关系、pair decision 与字段级 claims；按固定优先级一次展示一个补充问题，保存回答、跳过或保留不确定状态。
- 建立 53 条本地真实 HTTP smoke 断言及独立规则回归，覆盖 Mock 整理、馆藏 CRUD、人工确认边界、证据区间、编辑后证据失效、馆藏/考古备份恢复与 Demo 预置数据保护。

## 一行版

独立开发本地优先的 AI 辅助记忆策展工具，以 Node.js + SQLite 保存原文、结构化展品和可追踪整理流程，实现字段加权检索、引用式讲解及带原文字符区间的多版本记忆对照；事件关系由用户确认或解除，并通过 53 条本地 HTTP 断言验证核心闭环、备份恢复和 Demo 保护。

## 面试口述版

这个项目解决的不是“让 AI 改写一段文字”，而是私人记忆如何长期保存、重新找到，以及怎样处理同一往事在不同时间留下的多个版本。

用户先写原始记忆。系统通过一次模型调用生成结构化草稿；没有 Key 时使用本地规则回退。为了让处理过程可以回看，我把结果组织成档案提取、策展标注和草稿生成三个阶段并保存处理快照，但不把它描述成三个独立 Agent。草稿只有在用户确认后才进入馆藏，分析轨迹会先行保存。

检索会按标题、正文、人物、地点、标签和情绪加权。讲解时，系统把 Top-K 展品作为模型上下文，并把同批检索来源和命中原因交给用户核对；当前没有把提示词约束包装成程序级的逐句引用保证。

记忆考古部分不调用模型，而是使用确定性规则。当两段记录出现共同人物、地点或日期时，“记忆航线”只给出关联候选和理由，不判断它们一定属于同一事件。“时光拼图”比较日期、地点、人物、标签和情绪等字段；展示的字段证据必须带原文摘录和字符区间，缺失不会被当作矛盾。只有用户确认后才建立版本组，也可以解除分组，两个原文始终分别保留。

公开 Demo 使用四件示例展品和临时 SQLite，不部署开发者的私人馆藏或真实模型密钥。预置展品、导入、删除和清空已被保护；访客新增的示例内容仍可能在同一临时实例中被其他访客看到，因此它只用于体验，不是会话隔离环境。

## 可主动说明的工程取舍

- 选择 Node 原生 HTTP 和 Vanilla JS：项目规模不需要框架，依赖更少，面试时可以直接讲清数据流。
- 选择 SQLite：适合个人、本地优先场景；公开 Demo 的 `/tmp` 只用于体验，不伪装成生产持久化方案。
- 当前“语义检索”是词汇扩展和字段加权，不声称使用向量数据库。
- 当前记忆考古只验证摘录能否由 `rawContent.slice(start, end)` 复核，不验证记忆真假或任意自然语言推断。
- “补一块拼图”使用固定字段优先级，不声称进行了信息增益计算；回答作为独立记录保存，不自动改写原文。
- 馆藏与记忆考古备份不包含 `agent_runs / agent_steps / agent_events`，恢复时不会伪造旧运行轨迹。
